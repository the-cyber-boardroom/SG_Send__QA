## 1. The Three Test Layers: Scope, Focus, and Objective

SG/Send QA maintains **three distinct test layers** that complement each other. They are not alternatives — each covers ground the others cannot.

```
┌─────────────────────────────────────────────────────────────────────────┐
│  tests/unit/          Unit Tests                                        │
│  ─────────────────────────────────────────────────────────────────────  │
│  • Tests the QA framework itself (sg_send_qa/browser/, utils/, etc.)   │
│  • Pure Python — no browser, no server, no network                     │
│  • Fast: runs in seconds                                                │
│  • Scope: QA code quality gate                                         │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  tests/integration/   Integration Tests (NEW)                           │
│  ─────────────────────────────────────────────────────────────────────  │
│  • Tests the QA framework's browser interactions against the local      │
│    in-memory SG/Send stack (harness-managed lifecycle)                 │
│  • Focus: does the framework work? Shadow DOM, state capture, etc.     │
│  • Ordered tests within a class (shared browser session)               │
│  • Target: local stack only (no live deployments yet)                  │
│  • Uses: SG_Send__Browser__Test_Harness + SG_Send__Browser__Pages      │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  tests/qa/v030/       QA Acceptance Tests (v030 suite)                 │
│  ─────────────────────────────────────────────────────────────────────  │
│  • Tests the full functionality of the SG/Send site                    │
│  • Each test = one documented use-case with screenshots                │
│  • Produces the sg_send_qa__site documentation (living docs)           │
│  • Independent tests (pytest, fresh page per test)                     │
│  • Priority-tiered: P0 blockers / P1 sign-off / P2 informational       │
│  • Target: local stack now → live deployments (dev/main/prod) soon     │
│  • Uses: pytest fixtures from conftest.py + SG_Send__Browser__Pages    │
└─────────────────────────────────────────────────────────────────────────┘
```

### Why Keep Both Integration and v030?

| Dimension | Integration Tests | v030 QA Acceptance |
|---|---|---|
| **Primary question** | Does the framework work? | Does the product work? |
| **Test independence** | Ordered within a class (shared state) | Each test is independent |
| **State management** | Persistent harness, reusable page | Fresh page per test |
| **Output** | Pass/fail with framework diagnostics | Screenshots + structured metadata → documentation site |
| **Priority system** | All-or-nothing (CI gate) | P0/P1/P2 tiered gates |
| **Documentation** | None (framework tests) | Yes — every test produces a page for sg_send_qa__site |
| **Target** | Local stack only | Local stack + live lambda (dev/main/prod) |
| **Shadow DOM strategy** | SG_Send__Browser__Pages primitives | SG_Send__Browser__Pages primitives (refactored) |

They serve different masters. Integration tests ask "is the test tooling working?" QA acceptance tests ask "is the product working?" Both must pass, for different reasons.

## 3. Architecture: The Page Object + State Machine Pattern

Following the dev brief (`v0.16.61__dev-brief__playwright-optimization.md`) and arch brief (`v0.16.61__arch-brief__state-machines-ontologies.md`):

### 3.1 Page Object Classes

Each significant page in SG/Send gets a Python class that:
- Encapsulates all selectors and DOM interactions for that page
- Exposes named action methods (not raw locators)
- Provides `extract_page_model()` returning a typed dataclass

```
sg_send_qa/browser/
├── SG_Send__Browser__Pages.py       ← existing: routes + primitives (Layer 0)
├── pages/
│   ├── Upload_Page.py               ← new: upload wizard page object
│   ├── Gallery_Page.py              ← new: gallery view page object
│   ├── Browse_Page.py               ← new: browse/folder view page object
│   ├── Viewer_Page.py               ← new: single file viewer page object
│   └── Download_Page.py             ← new: download + key entry page object
└── schemas/
    ├── Schema__Upload_Page.py       ← typed state capture for upload page
    ├── Schema__Gallery_Page.py      ← typed state capture for gallery page
    ├── Schema__Browse_Page.py
    ├── Schema__Viewer_Page.py
    └── Schema__Download_Page.py
```

### 3.2 Page State as Typed Schema (JSON-serialisable)

```python
@dataclass
class Schema__Upload_Page:
    step: str                        # 'idle' | 'file-ready' | 'choosing-delivery' |
                                     # 'choosing-share' | 'confirming' | 'uploading' | 'complete'
    filename: str | None
    filesize_bytes: int | None
    delivery_mode: str | None        # 'file' | 'text'
    share_mode: str | None           # 'combined' | 'token' | 'separate'
    access_gate_visible: bool
    next_button_visible: bool
    next_button_label: str | None
    done_link: str | None            # populated in 'complete' state
    captured_at: str                 # ISO 8601
```

Tests assert against schema, not DOM:
```python
model = upload_page.extract_page_model()
assert model.step == 'complete'
assert model.done_link is not None
assert '#' in model.done_link
```

### 3.3 Upload Wizard State Machine

The upload wizard is a linear state machine with branching:

```
idle
  │ (file selected)
file-ready
  │ (Next clicked)
choosing-delivery        ← only for multi-file / zip
  │ (Next clicked or skipped)
choosing-share
  │ (mode card clicked → auto-advances)
confirming
  │ (Encrypt & Upload clicked)
uploading                ← encrypting → creating → uploading → completing
  │ (complete)
complete                 ← done step: link/token shown
```

### 3.4 localStorage as Inter-Test State Channel

Upload tests write the transfer ID and key to localStorage:
```python
self.js_evaluate(f"""
  localStorage.setItem('qa-transfer-id', '{transfer_id}');
  localStorage.setItem('qa-transfer-key', '{key_b64}');
""")
```

Download tests read it back:
```python
transfer_id = self.js_evaluate("localStorage.getItem('qa-transfer-id')")
key_b64     = self.js_evaluate("localStorage.getItem('qa-transfer-key')")
```

This enables linked upload → download test sequences without Python-level state passing.

## 4. Implementation Plan

### Phase 1: Fix Integration Tests (`tests/integration/browser/`)

**Goal:** All integration browser tests pass, covering all major page interactions.

#### 1a — Add `transfer_helper()` to Harness

The harness lacks a `transfer_helper()` method — blocking all download-path tests. Add it:

```python
# SG_Send__Browser__Test_Harness.py
def transfer_helper(self):
    from sg_send_qa.utils.QA_Transfer_Helper import QA_Transfer_Helper
    return QA_Transfer_Helper(
        api_url=self.api_server.server_url(),
        access_token=self.test_objs.access_token,
    )
```

#### 1b — Uncomment + Complete Workflow Tests

`test_SG_Send__Browser__Pages__Workflow.py` — implement the upload-then-download round trip using localStorage handoff. This is the keystone integration test.

#### 1c — Complete Access Gate Tests

`test_SG_Send__Browser__Pages__Access_Gate.py` — test the gate with wrong token, correct token, and pre-set token via localStorage.

#### 1d — Complete Download Page Tests

`test_SG_Send__Browser__Pages__Download.py` — test browse, gallery, viewer, and separate-key download using transfers created via `transfer_helper()`.

#### 1e — Add Page Object Classes

Create `sg_send_qa/browser/pages/` with page classes as per the dev brief. Start with `Upload_Page` and `Download_Page` since these are the core workflows.

#### 1f — Unit Tests for Framework

For every new framework class, companion unit tests in `tests/unit/browser/`. These test the class structure, schema serialisation, and state machine logic without a browser.

**Run after each sub-phase:**
```bash
source /tmp/venv312/bin/activate && python -m pytest tests/integration/browser/ -v --tb=short
source /tmp/venv312/bin/activate && python -m pytest tests/unit/browser/ -v --tb=short
```

### Phase 2: Refactor v030 Tests

**Goal:** v030 tests use `SG_Send__Browser__Pages` primitives instead of raw Playwright selectors. Tests remain pytest-based, independent, and screenshot-producing.

#### 2a — Wire `SG_Send__Browser__Pages` into v030 conftest

Add a session-scoped `sg_send` fixture that provides a `SG_Send__Browser__Pages` instance wired to the same UI server as the `page` fixture:

```python
@pytest.fixture(scope="session")
def sg_send(ui_server):
    from sg_send_qa.browser.SG_Send__Browser__Pages import SG_Send__Browser__Pages
    # wire target_port from ui_server URL
    port = int(ui_server.split(":")[-1].rstrip("/"))
    instance = SG_Send__Browser__Pages(headless=True, target_port=port)
    yield instance
    instance.qa_browser().stop()
```

**BUT** v030 tests use the pytest `page` fixture (fresh page per test). The `sg_send` fixture would use its own browser. These need to be reconciled: either:
- v030 tests switch from `page` fixture to `sg_send` methods (preferred — simpler)
- OR create a `SG_Send__Browser__Pages` that accepts an external `page` (more complex)

**Decision:** v030 tests will gradually move to using `sg_send` methods directly, retiring the raw `page` fixture as tests are migrated.

#### 2b — Replace Raw Selectors in P0 Tests First

Starting with the most critical (and formerly failing) tests:

| Test file | Current approach | New approach |
|---|---|---|
| `p0__upload__single_file` | `page.locator("#file-input")` | `sg_send.upload__set_file()` |
| `p0__combined_link` | Raw selectors + manual waits | `sg_send.upload__*` methods |
| `p0__friendly_token` | Same | Same |
| `p0__separate_key` | `input[placeholder*='key']` | `sg_send.download__enter_key()` |

#### 2c — Replace `networkidle` and `wait_for_timeout` with Event-Based Waits

Replace:
```python
page.wait_for_load_state("networkidle")
page.wait_for_timeout(3000)
```

With:
```python
sg_send.wait_for_upload_state("complete")   # or specific component waits
```

#### 2d — Use `visible_text()` and `inner_text()` Consistently

Replace all `page.text_content("body")` with `sg_send.visible_text()` (which calls `inner_text("body")`) to eliminate the false "error" false positives from inline script tags.

#### 2e — Restore CI Pipeline (Staged)

**Stage 1:** Re-enable `qa-acceptance-tests.yml` for **P0 only**, using refactored P0 tests:
```yaml
- name: Run P0 acceptance tests
  run: python -m pytest tests/qa/v030/ -m "p0" --tb=short
```

**Stage 2:** After P0 stable for 2 runs, add P1 with `continue-on-error: true`.

**Stage 3:** Wire the QA acceptance tests to trigger after SG/Send deployments to dev/main/prod lambdas (the acceptance criteria for live deployment gating).

## 5. Future: Live Deployment Testing

The v030 tests are currently run against the local in-memory stack only. The plan is to also run them against live deployments:

```
Local stack (current):
  tests/qa/v030/ → SG_Send__Browser__Test_Harness (in-memory API + local UI)

Live deployments (future):
  tests/qa/v030/ → configure SG_Send__Browser__Pages to point to:
    - https://dev.send.sgraph.ai  (dev lambda, post-deploy trigger)
    - https://send.sgraph.ai      (main/prod lambda, manual trigger or post-release)
```

The key change is in how `SG_Send__Browser__Pages.target_server` and `target_port` are configured. For live deployments, there is no local API server — the browser talks directly to the live endpoint. The access token would come from environment secrets rather than the in-memory test server.

The `conftest.py` will support a `--target` flag:
```bash
pytest tests/qa/v030/ --target=live --target-url=https://send.sgraph.ai -m "p0"
```
