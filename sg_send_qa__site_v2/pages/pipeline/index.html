How the QA Site Works

This page documents how qa.send.sgraph.ai is built, updated, and deployed. The site is living documentation — every page is backed by automated Playwright browser tests that capture screenshots as evidence.

The Pipeline

Every push to the repo triggers the full pipeline:

1

Tests run and capture screenshots

Playwright browser tests in tests/qa/v030/ run against the real SG/Send platform via headless Chromium. Each test calls screenshots.capture(page, name, description) to save a PNG alongside a _metadata.json file.

2

Screenshot diffing

New screenshots are compared against the previous commit using Pillow. If the pixel difference is under 1%, the file is reverted — preventing noisy commits from sub-pixel rendering differences. If it exceeds 1%, the new screenshot is kept.

3

Documentation is generated

generate_docs.py walks the use-cases directory. For each folder with a _metadata.json, it scaffolds a documentation page containing the test description, screenshot gallery, and test source code. Hand-crafted pages (like this one) are never overwritten.

4

Changes are committed to git

Updated screenshots and generated documentation pages are committed back to the branch automatically. Git becomes the historical record: every version of every page and screenshot is preserved.

5

Site is deployed to S3 / CloudFront

deploy_static_site.py uploads the committed site files to S3 and invalidates the CloudFront cache. The new version of qa.send.sgraph.ai is live within minutes of the push.

What Each Test Produces

Every browser test produces two outputs:

  • A pass/fail result — the CI pipeline goes green or red
  • A set of annotated screenshots — these become the documentation page you are reading now

A screenshot entry pairs a visual state with the code that produced it: 

Action:  page.drop_file("test-photo.png")
Code:    upload_page.drop_file(test_files["photo"])
         upload_page.wait_for_state("file-ready")
         screenshot("02_file_selected")

Local Development

To preview the v2 site locally:

./scripts/qa__run-locally.sh
# → http://localhost:10064/

No Ruby, no Jekyll, no build step. Screenshots load from the live site via full URLs.

To run tests and capture screenshots locally:

python -m pytest tests/qa/v030/test__access_gate.py -v
python -m sg_send_qa.cli.generate_docs_v2

Adding a New Test

Option A: Auto-scaffolded (from test output)

  1. Write a test in tests/qa/v030/ that uses the screenshots fixture
  2. The fixture creates the folder structure automatically
  3. generate_docs.py creates a starter HTML page on the next CI run
  4. CI commits and deploys

Option B: Hand-crafted (richer documentation)

  1. Create sg_send_qa__site_v2/pages/use-cases/{group}/{name}/index.html
  2. Use the template in sg_send_qa__site_v2/_common/templates/test-page.html
  3. Write your test to capture screenshots; update _metadata.json
  4. Add the use case to _data/qa_sidebar.json

Screenshot Naming Convention

Screenshots follow a consistent naming scheme:

screenshots/
  01_landing.png              ← NN_description (sequential, snake_case)
  02_token_entered.png
  03_upload_ready.png
  _metadata.json              ← test docs + screenshot descriptions

The NN_ prefix sets the display order. The description becomes the screenshot title (underscores → spaces, title-cased).
