## Status: ⛔ P0 — v0.20.30 still broken

The fix in v0.20.30 (commit `483f81ad`) has the correct logic but was applied with a merge
artifact that leaves orphaned code after the function's closing `}`. This causes a
`SyntaxError` when Chromium parses `send-download.js`, making the entire component fail
silently — the browse tree never renders, same symptom as v0.20.29.

---

## Root Cause

The patch in commit `483f81ad` correctly rewrites `_extractMetadata`. However, the diff did
not cleanly delete the old loop body. The old format's scanner loop (`for (let j...`)
closing lines were left in the file as orphaned code **after** the new function's closing
`}`:

```
// line 262 — correct end of _extractMetadata
        } catch (_) { return { metadata: null, content: buf }; }
    }                                                            ← function closes here

// lines 263–271 — ORPHANED (should have been deleted by the patch)
            if (bytes[j] === 0x7B) depth++;
            else if (bytes[j] === 0x7D) { if (--depth === 0) { jsonEnd = j + 1; break; } }
        }
        if (jsonEnd < 0) return { metadata: null, content: buf };
        try {
            const metaStr = new TextDecoder().decode(bytes.slice(jsonOffset, jsonEnd));
            return { metadata: JSON.parse(metaStr), content: buf.slice(jsonEnd) };
        } catch (_) { return { metadata: null, content: buf }; }
    }
```

This is syntactically invalid inside a class body — a bare statement block without a method
declaration. Chromium throws:

```
SyntaxError: Unexpected token '['
    at send-download.js:263
```

The `SendDownload` class fails to define. All transfers break: `zipTree=0`, browse tree
never renders. Identical symptoms to v0.20.29.
