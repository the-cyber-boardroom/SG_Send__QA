### 1. Gallery page is NOT patched for new `__gallery__` folder naming

**Severity: P1 (potential)**

The `upload-folder-v031.js` + `send-gallery-v031.js` change the gallery metadata folder name from `_gallery.{16-char}` (v0.3.0) to `__gallery__{8-char-hash}` (v0.3.1).

**Problem**: `send-gallery-v031.js` is only loaded on the `/en-gb/browse/` page. The `/en-gb/gallery/` page has **no v0.3.1 overlay**. The changelog confirms:

> Pages with v0.3.1 Active:
> - `/en-gb/gallery/` — v0.3.0, None (no v0.3.1 overlay yet)

**Impact**: Users who upload via v0.3.1 and then navigate to `/en-gb/gallery/#hash` may see the gallery metadata folder as a visible folder item (since the gallery page doesn't know about the new naming convention and won't filter it out).

**QA request**: Please confirm whether this is:
a) Intentional (gallery page upgrade is a separate sprint), or
b) A bug (gallery page should also load `send-gallery-v031.js`)

If intentional, add a note to the changelog. If a bug, assign a BRW ID (suggest BRW-016) and add a v0.3.1 patch for the gallery page.

### 2. `sg-layout` loaded from external CDN — no local fallback

**Severity: P1 infrastructure**

The v0.3.1 browse page loads `sg-layout` from:
```
https://dev.tools.sgraph.ai/core/sg-layout/v0.1.0/sg-layout.js
```

This is an **undocumented hard dependency on an external CDN**. QA discovered this when tests failed with `net::ERR_FAILED` because the CDN was not reachable in the test environment.

**Workaround applied by QA**: Downloaded `sg-layout-v010.js` and `sg-layout-events-v010.js` locally. Added CDN URL patching in `QA_UI_Server.py` to replace the CDN URL with a local `/_vendor/sg-layout.js` path.

**Risks**:
1. CDN downtime = browse view completely broken for all users
2. `sg-layout.js` imports `sg-layout-events.js` as a relative module — if the CDN serves one but not the other, the component fails silently
3. The CDN URL pattern uses the path `dev.tools.sgraph.ai/core/sg-layout/v0.1.0/` — is `v0.1.0` pinned or could it change without notice?

**QA request**:
- Bundle `sg-layout.js` with the UI package (include it in the installed `sgraph_ai_app_send__ui__user` package under `_common/js/`)
- OR serve it from the same origin as the app (self-hosted CDN)
- OR add a clearly documented fallback in the HTML for CDN failure
- Update `_copy_ui_layer` in `QA_UI_Server.py` once bundled so we don't need the local copy workaround

### 4. Multiple `.sb-file__view-source` buttons — hidden/visible duality

**Severity: P3 (documentation/awareness)**

When multiple files are open in tabs, there is one `.sb-file__view-source` button per open file. Only the active tab's button is visible (the others have `display:none` or similar CSS that Playwright considers "hidden").

**QA observation**: `page.locator(".sb-file__view-source").first` selects the FIRST button in DOM order, which may be the HIDDEN button for an earlier tab. Tests must use `:visible` pseudo-class:
```python
source_btn = page.locator(".sb-file__view-source:visible").first
```

**This is correct behavior** from a UX perspective. Documenting it so other test authors know.

**QA request**: None required — this is documented here for future test authors.

### 5. `data-ready` timing with overlay JS loading

**Severity: P2 investigation**

The `body[data-ready]` attribute is set by SG/Send bootstrap when the app shell is ready. In v0.3.1, the browse page loads 4 additional overlay JS files (`send-browse-v031.js`, `send-browse-v031.css`, `markdown-parser-v031.js`, `send-gallery-v031.js`).

**Observation**: After `body[data-ready]` is set, the overlay patches may still be in progress. Tests that click a file immediately after `body[data-ready]` + `1.5s` wait sometimes find that the `.sb-file__view-source` button is attached (in DOM) but hidden, because the file content hasn't finished rendering.

**Root cause theory**: The button is created by the component when a file is opened, but shown only after async markdown rendering completes.

**QA workaround**: Added `page.wait_for_function(...)` polling for `button.offsetParent !== null && button.getBoundingClientRect().height > 0` before asserting button visibility.

**QA request**: If there's a DOM signal the overlay patches can emit when content is fully rendered (e.g. a custom event or an attribute on the file panel), please share it. This would let QA write more deterministic tests without polling.

## ✅ RESOLVED (31 March 2026) — P0 Regression in send-download.js

**Status: FIXED. All PLR tests passing. v0.3.1 cleared for release.**

### Bug: Deep-link fix strips the encryption key from the hash

The commit "fix send-download.js (v0.3.0): strip deep-link path from hash before token parse"
introduced a P0 regression that **breaks every non-friendly-token browse transfer**.

**Broken code in `send-download.js` `_parseUrl()`:**

```javascript
// BROKEN — strips FIRST '/' which for #transferId/key is the key separator
const hash = _fullHash.includes('/')
               ? _fullHash.slice(0, _fullHash.indexOf('/'))  // discards encryption key!
               : _fullHash;
// ...
const i = hash.indexOf('/');   // -1, key already gone
if (i > 0) { ... }             // never reached
else { this.transferId = hash; // hashKey never set }
```

For a URL like `/browse/#abc123/encryption_key_here`:
1. `_fullHash = "abc123/encryption_key_here"`
2. `hash = "abc123"` ← **key discarded!**
3. `this.hashKey` is never set → decryption fails silently
4. `sd.state = 'ready'`, `sd.zipTree = []` → browse tree never renders

**Friendly-token URLs are unaffected** (`#word-word-NNNN` has no `/`), but all standard `#transferId/encryptionKey` URLs fail.

**Correct fix:**

```javascript
const firstSlash = _fullHash.indexOf('/');
const firstPart  = firstSlash > 0 ? _fullHash.slice(0, firstSlash) : _fullHash;
let hash;
if (firstSlash < 0) {
    hash = _fullHash;  // no slash — no path suffix
} else if (typeof FriendlyCrypto !== 'undefined' && FriendlyCrypto.isFriendlyToken(firstPart)) {
    hash = firstPart;  // friendly token: strip path after FIRST slash
} else {
    // Regular #tid/key[/path]: strip path suffix after SECOND slash (keep key)
    const secondSlash = _fullHash.indexOf('/', firstSlash + 1);
    hash = secondSlash > 0 ? _fullHash.slice(0, secondSlash) : _fullHash;
}
```

**Send team fix (31 March 2026):** Changed deep-link separator from `/` to `|`. New format: `#tid/key|path/to/file`. The `|` character never appears in transfer IDs, keys, or file paths, making it unambiguous. QA updated PLR tests to use `|` separator. All 13 PLR tests now pass.

**QA tracking:**
- `tests/qa/v031/bugs/test__bug__send_download_deeplink_strips_key.py` — DELETED (both bug tests flipped to FAIL on 31 March 2026, confirming fix landed correctly)

**Severity: P2** (revised down from P1 — see note below)

BRW-019 (`hashchange` listener on `SendDownload`) does **not work** in the current implementation.

**Root cause** (confirmed by QA diagnostics):

The `send-browse-v031.js` IIFE patches `SendDownload.prototype.connectedCallback` to register a hashchange listener. But the `<send-download>` element in `browse/index.html` (line 36) is parsed and upgraded **before** the `<script>` tags (line 49+) execute.

Execution order:
1. HTML parsed to line 36 → `<send-download>` element in DOM (not yet upgraded)
2. `send-download.js` runs → `customElements.define()` called → element upgraded → **`connectedCallback` fires (UNPATCHED)**
3. `send-browse-v031.js` runs → patches `SendDownload.prototype.connectedCallback` (too late)

Result: `sd._v031HashHandler` is never set. External hash changes (browser back/forward, programmatic navigation) are not handled.

**Severity revised to P2 (31 March 2026):** The entry form "Decrypt & Download" button was confirmed to work correctly — it calls the transfer load logic directly rather than relying on `hashchange`. BRW-019 does not affect the primary user journey (enter token → browse opens). It only affects navigation scenarios where the hash changes *after* initial load from an external source (e.g., programmatic navigation, browser history traversal).

**Fix** (in `send-browse-v031.js`, add at the end of the BRW-019 IIFE):

```javascript
// Apply to already-connected instances (connectedCallback already fired)
document.querySelectorAll('send-download').forEach(function(el) {
    if (!el._v031HashHandler) {
        el._v031HashHandler = function() {
            el.transferId = null; el.hashKey = null;
            el.tokenName = null; el._friendlyToken = null;
            el._parseUrl();
            if (!el.transferId) { el.state = 'entry'; el.render(); el._setupEntryListeners(); }
            else { el.state = 'loading'; el.render(); el._loadTransferInfo(); }
        };
        window.addEventListener('hashchange', el._v031HashHandler);
    }
});
```

**QA tracking**: `tests/qa/v031/bugs/test__bug__brw019_hashchange_handler_not_registered.py` — passes while bug exists, fails when fixed.
