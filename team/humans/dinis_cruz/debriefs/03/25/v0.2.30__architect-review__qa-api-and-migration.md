The brief is structurally sound. The three tasks are correctly sequenced, and the dependency graph (schemas → API + migration) is accurate. ## Task 1: Schema Wiring — Architecture Sign-off

### Assessment

The approach is correct. `extract__*()` methods belong on `SG_Send__Browser__Pages` as read-only snapshot methods. They compose existing `self.js()` calls and return a `Type_Safe` schema. No new JS queries are needed — every value is accessible through the existing query layer.

### Exact Selectors (confirmed from codebase)

The selectors in the brief can be confirmed directly from `SG_Send__Browser__Pages.py`:

**Upload page:**
- `state` → `self.upload_state()` → `self.js().light_property("send-upload", "_state")`
- `file_name` → `self.js().text("upload-step-select", "#file-name")` *(selector to verify against live DOM)*
- `share_link` → `self.js().text("upload-step-done", "#combined-link")` ✓ used in `upload__get_combined_link()`
- `friendly_token` → `self.js().text("upload-step-done", "#simple-token")` ✓ used in `upload__get_friendly_token()`
- `is_gate_visible` → `self.is_access_gate_visible()` ✓

**Download page:**
- `state` → `self.download_state()` → `self.js().light_property("send-download", "state")`
- `is_key_input_visible` → `self.js().light_visible("#key-input")` ✓ used in `download__enter_key()`
- `content_text` → `self.visible_text()` → `self.js().light_text("body")`
- `error_message` → `self.js().light_text(".error-message")` *(selector to verify — may need DOM inspection)*
- `transfer_id` → extract from `self.url()` (parse hash fragment after `#`)

**Browse/Gallery/Viewer pages:** schema fields are mostly `visible_text()` and state reads. Selectors for `file_count` (gallery), `folder_tree` (browse), and `content_type` (viewer) need DOM inspection before implementation.

### Approved Pattern

```python
def extract__upload_page(self) -> Schema__Upload_Page:
    return Schema__Upload_Page(
        state           = self.upload_state()                                    or '',
        file_name       = self.js().text("upload-step-select", "#file-name")     or '',
        share_link      = self.js().text("upload-step-done", "#combined-link")   or '',
        friendly_token  = self.js().text("upload-step-done", "#simple-token")    or '',
        is_gate_visible = self.is_access_gate_visible()                              ,
    )
```

No raw `page.evaluate()`. No string interpolation. Everything through `self.js()`. This is the correct pattern.

### Amendment

The `transfer_id` field on `Schema__Download_Page` should be populated from the URL hash, not from the DOM:

```python
def _transfer_id_from_url(self) -> str:
    fragment = self.url().split('#')[-1] if '#' in self.url() else ''
    return fragment.split('/')[0] if fragment else ''
```

This is a private helper — not a public API, just used by `extract__download_page()`.

## Task 2: QA API — Architecture Sign-off

### Assessment

The 4-layer design is well-structured and the stateless-per-request model is the correct choice for Lambda compatibility. The `Schema__Capture_Config` pattern is clean — caller declares what they want, server returns exactly that.

### Approved File Structure

```
sg_send_qa/api/
├── QA_API__Server.py           — FastAPI app, route registration, health endpoint
├── QA_API__Runner.py           — execution engine: local/live dispatch, captures, teardown
├── QA_API__Session.py          — browser session lifecycle (create, inject trace ID, teardown)
├── Schema__Capture_Config.py   — Type_Safe: 12 capture flags
├── Schema__QA_Request.py       — Type_Safe: target, access_token, mode, trace_id, capture, options
├── Schema__QA_Response.py      — Type_Safe: status, duration_ms, + optional captured fields
└── routes/
    ├── routes__browser.py      — Layer 0
    ├── routes__page.py         — Layer 1
    ├── routes__upload.py       — Layer 2
    ├── routes__download.py     — Layer 2
    └── routes__workflow.py     — Layer 3
```

Workflow files (`persona_a`, etc.) belong in `routes__workflow.py` initially, not a separate `workflows/` subpackage — keep it flat until there are enough workflows to justify splitting.

### Key Design Decisions (signed off)

1. **All endpoints are POST** except `GET /api/qa/health` — correct.
2. **Stateless per request** — no browser pool, no session cache — correct.
3. **`Schema__Capture_Config` as a `Type_Safe` field on `Schema__QA_Request`** — correct. Use `None` default, populate with `Schema__Capture_Config()` defaults in `QA_API__Runner` if not provided.
4. **Local mode** uses `SG_Send__Browser__Test_Harness` — correct. This gives us the full server lifecycle.
5. **Live mode** takes `target` URL + `access_token` — correct. No `Test_Harness`, just `SG_Send__Browser__Pages` pointed at the live URL.

### Amendment: `QA_API__Runner` is the key class

The runner needs to be the single place where:
- local vs live dispatch is decided
- trace ID is injected
- capture collectors are started/stopped
- browser is torn down after every request

Route handlers should be thin wrappers:
```python
@router.post("/api/upload/combined")
def upload_combined(req: Schema__Upload__Combined_Request):
    return QA_API__Runner(request=req).run__upload_combined()
```

No business logic in route handlers. All execution in the runner.

### Open Questions for Task 2

1. **`Schema__Upload__Combined_Request`** — do we create per-endpoint request schemas (inheriting `Schema__QA_Request` + endpoint-specific fields) or use a generic `options: dict`? Recommendation: per-endpoint schemas (Type_Safe, explicit fields) — avoids `dict` which loses type safety.
2. **Where do screenshots go in the response?** The brief says `"screenshots": ["base64..."]` — confirm: base64-encoded PNG bytes, array (one per screenshot taken during the workflow), not a single value.
3. **`prometheus` format** — confirm: standard Prometheus exposition format (`text/plain`) embedded in JSON as a string field, not a separate content-type endpoint.

## Task 3: v030 Migration — Architecture Sign-off

### Assessment

The phased approach is correct. Phase 0 (TestCase conversion) is the right first step — it decouples test lifecycle from conftest fixtures and makes shared browser state explicit. The target pattern matches the integration tests exactly, which is the right convergence point.

### Approved Migration Pattern

```python
class test_Single_File_Upload(TestCase):
    @classmethod
    def setUpClass(cls):
        cls.harness = SG_Send__Browser__Test_Harness().headless(True).setup()
        cls.sg_send = cls.harness.sg_send
        with cls.harness as _:
            cls.access_token = _.set_access_token()

    @classmethod
    def tearDownClass(cls):
        cls.harness.teardown()
```

The `with cls.harness as _` pattern for token setup needs to be verified — check how `SG_Send__Browser__Test_Harness` implements `__enter__`/`__exit__` before committing to this.

### What Gets Removed in Phase 4

These conftest items become dead code once all tests are migrated:
- `subprocess.Popen` server startup
- `while time.time() < deadline` polling loop
- `playwright_instance`, `browser`, `browser_context`, `page` fixtures
- `ui_url`, `send_server` fixtures (replaced by harness)
- `browser_helpers.py` entirely

Keep: `transfer_helper` fixture (API-level test setup), marker registration.

### Phasing Risk

The main risk is **partial migration coexistence**: migrated `TestCase` tests use the harness (which starts its own servers on fresh ports), while non-migrated tests use conftest fixtures (which start servers on fixed ports). If the tests are in the same pytest run, port conflicts are possible.

**Mitigation:** Use `target_port=0` (or dynamic port allocation) in the harness — confirm this is how it works, or add it before migration starts.
