## What Changed in This Update

The original brief described the CLI's purpose and command structure correctly. This update adds:

1. **Class hierarchy** — Type_Safe class structure matching the SG_Send__CLI patterns (`CLI__Main` holds command group classes as typed attributes)
2. **Safe_Str types** — domain-specific types for the QA CLI, matching the 35+ types in `sgit_ai/safe_types/`
3. **Schemas for session and run data** — Type_Safe pure data classes for all CLI domain objects
4. **The `cmd_*` pattern** — each command is a method on a Type_Safe class, testable with `SimpleNamespace` args
5. **State machine commands** — the state machine infrastructure we just built needs CLI exposure
6. **`--capture` flag** — connects to the existing `Schema__Capture_Config`
7. **Concrete file layout** — every file that needs creating, with its purpose

## What We Need

A command-line interface for the QA automation that separates triggering from execution. The CLI drives the FastAPI service. The FastAPI service drives Playwright. The CLI can invoke the FastAPI locally, on a Lambda, on an EC2 instance, or anywhere else the service is deployed.

```
CLI (triggers)
  │
  ├── --target local     → calls FastAPI on localhost
  ├── --target lambda    → calls FastAPI on Lambda function URL
  ├── --target ec2       → calls FastAPI on EC2 instance
  │
  └── FastAPI (executes)
        │
        └── Playwright (browser automation)
              │
              └── Target website (dev, main, prod)
```

This separation means the CLI is lightweight (makes HTTP calls) and the execution environment is flexible (runs wherever the FastAPI is deployed).

## Class Hierarchy

Following the SG_Send__CLI pattern exactly: `CLI__Main` is the entry point, holds command group classes as Type_Safe attributes, builds the argparse parser, and dispatches to `cmd_*` methods.

```python
from osbot_utils.type_safe.Type_Safe import Type_Safe

class CLI__QA__Main(Type_Safe):
    run          : CLI__QA__Run             # sg-qa run <workflow>
    session      : CLI__QA__Session         # sg-qa session start/stop/list/view
    traffic      : CLI__QA__Traffic         # sg-qa traffic --users N
    state_machine: CLI__QA__State_Machine   # sg-qa sm export/mermaid/coverage/diff
    config       : CLI__QA__Config          # sg-qa config set/get
    report       : CLI__QA__Report          # sg-qa report <session-id>
    export       : CLI__QA__Export          # sg-qa export --vault/--s3

    def build_parser(self) -> argparse.ArgumentParser: ...
    def run(self, argv=None): ...
```

Each command group class follows the SG_Send__CLI `CLI__Vault` pattern: Type_Safe, holds its dependencies as typed attributes, exposes `cmd_*` methods that receive `argparse.Namespace` args.

```python
class CLI__QA__Run(Type_Safe):
    endpoint_config : Schema__QA_Endpoint_Config

    def cmd_run(self, args):           ...    # sg-qa run <workflow>
    def cmd_list(self, args):          ...    # sg-qa run --list

class CLI__QA__Session(Type_Safe):
```

## Safe_Str Types

Following the SG_Send__CLI's `sgit_ai/safe_types/` pattern: one file per type, domain-specific validation. These live in `sg_send_qa/cli/safe_types/`.

```python
from osbot_utils.type_safe.primitives.domains.web.safe_str.Safe_Str__Url import Safe_Str__Url
class Safe_Str__API_Endpoint(Safe_Str__Url): pass        # e.g. 'https://qa-lambda.sgraph.ai'

# sg_send_qa/cli/safe_types/Safe_Str__API_Key.py
from osbot_utils.type_safe.primitives.domains.identifiers.safe_str.Safe_Str__Id import Safe_Str__Id
```

## Schemas

Pure data classes for all CLI domain objects. No methods (Rule 5). Serialisation is free via `.json()` / `.from_json()`.

```python
# sg_send_qa/cli/schemas/Schema__QA_Endpoint_Config.py
class Schema__QA_Endpoint_Config(Type_Safe):
    endpoint     : Safe_Str__API_Endpoint
    api_key      : Safe_Str__API_Key
    target       : Safe_Str__Target_Name = 'local'

# sg_send_qa/cli/schemas/Schema__QA_Session.py
class Schema__QA_Session(Type_Safe):
    session_id    : Safe_Str__Session_Id
    target        : Safe_Str__Target_Name
    endpoint      : Safe_Str__API_Endpoint
    started_at    : Timestamp_Now
    stopped_at    : Timestamp_Now
    run_count     : Safe_UInt
    failure_count : Safe_UInt
    runs          : List[Schema__QA_Run_Summary]

# sg_send_qa/cli/schemas/Schema__QA_Run_Summary.py
class Schema__QA_Run_Summary(Type_Safe):
    run_id        : Safe_Str__Id
    workflow      : Safe_Str__Workflow_Name
    status        : Literal['pass', 'fail', 'error']
    duration_ms   : Safe_UInt
    trace_id      : Safe_Str__Id
    screenshot_count : Safe_UInt
    transitions_observed : List[State_Transition]         # reuse from state_machines package

# sg_send_qa/cli/schemas/Schema__QA_Traffic_Config.py
class Schema__QA_Traffic_Config(Type_Safe):
    user_count    : Safe_UInt__User_Count
    workflow      : Safe_Str__Workflow_Name
    duration_secs : Safe_UInt__Duration_Secs
    mode          : Literal['browser', 'http'] = 'browser'
    mix           : bool                       = False     # random workflow mix
```

## Core Commands

### Health and status

```bash
sg-qa health                          # Check FastAPI service is reachable
sg-qa status                          # Current session info, target server, version
```

### Single workflow execution

```bash
sg-qa run upload-file                 # Run the file upload workflow
sg-qa run download-file               # Run the file download workflow
sg-qa run gallery-view                # Run the gallery view workflow
sg-qa run friendly-token              # Run the friendly token flow
sg-qa run combined-link               # Run the full upload + download
sg-qa run --list                      # List available workflows

# Run against a specific target
sg-qa run upload-file --target prod   # Run against production
sg-qa run upload-file --target dev    # Run against dev server
sg-qa run upload-file --target local  # Run against local server

# Control what data is captured (connects to Schema__Capture_Config)
sg-qa run upload-file --capture screenshots,video,browser-console
sg-qa run upload-file --capture minimal     # execution_result + performance_data only
sg-qa run upload-file --capture full        # everything on
```

The `--capture` flag maps directly to `Schema__Capture_Config` fields. The CLI parses the comma-separated values and sets the corresponding booleans:

```python
CAPTURE_PRESETS = {
    'minimal' : Schema__Capture_Config(execution_result=True, performance_data=True),
    'full'    : Schema__Capture_Config(execution_result=True, page_model=True,
                                       screenshots=True, video=True,
                                       browser_console=True, performance_data=True,
                                       prometheus=True, network_log=True),
}
```

### State machine commands

```bash
sg-qa sm export                       # Export JSON schemas to _data/state_machines/
sg-qa sm mermaid upload               # Print mermaid diagram for upload
sg-qa sm mermaid download             # Print mermaid diagram for download
sg-qa sm coverage <session-id>        # Compute transition coverage from session
sg-qa sm diff v0.2.35 v0.3.0         # Compare snapshots between versions
sg-qa sm validate                     # Validate all defined state machines (parse check)
```

These wrap the existing `State_Machine__Utils` methods and `export_schemas.py`. No new logic — CLI is a thin wrapper over existing infrastructure.

### Traffic generation

```bash
# Browser-based traffic (Playwright sessions)
sg-qa traffic --users 5 --workflow upload-file
sg-qa traffic --users 20 --workflow combined-link --target prod --duration 5m

# HTTP-only traffic (no browser, faster, cheaper)
sg-qa traffic --users 100 --mode http --target prod --duration 10m

# Random workflow mix
sg-qa traffic --users 10 --mix --target dev --duration 2m
```

### Session management

```bash
sg-qa session start                   # Start a new session (creates run folder)
sg-qa session stop                    # End session, package artifacts
sg-qa session list                    # List past sessions
sg-qa session view <id>               # View session results
```

### Data and reporting

```bash
sg-qa report <session-id>             # Generate a report from session data
sg-qa report --latest                 # Report from most recent session
sg-qa export <session-id> --vault     # Export session to a vault
sg-qa export <session-id> --s3        # Export session to S3
```

### Configuration

```bash
sg-qa config set endpoint http://localhost:8000     # Local
sg-qa config set endpoint https://qa-lambda.sgraph.ai  # Lambda
sg-qa config set api-key <key>                       # API key for auth
sg-qa config get endpoint                            # Show current endpoint
sg-qa config list                                    # Show all config
```

Or pass per-command:

```bash
sg-qa run upload-file --endpoint https://qa-lambda.sgraph.ai --api-key <key>
```

## Sessions and Run Capture

### The Session Concept

Every CLI invocation produces data. A session groups related invocations:

```bash
sg-qa session start
# Session abc123 started

sg-qa run upload-file
sg-qa run download-file
sg-qa run gallery-view

sg-qa session stop
# Session abc123 complete: 3 runs, 0 failures
```

### The Run Folder

Each session produces a folder containing all artifacts:

```
sessions/
  abc123/
    session.json            # Schema__QA_Session (metadata, config, timestamps)
    runs/
      001-upload-file/
        result.json         # Schema__QA_Run_Summary (success/failure, timing)
        metrics.prom        # Prometheus-formatted metrics
        snapshot.json       # State_Machine__Snapshot (transitions observed)
        screenshots/        # If --capture screenshots
        trace.json          # Playwright trace (if --capture video)
      002-download-file/
        result.json
        metrics.prom
        snapshot.json
      003-gallery-view/
        result.json
        metrics.prom
    summary.json            # Aggregated results, pass/fail counts
    coverage.json           # State machine coverage from all runs
    report.md               # Human-readable report (auto-generated)
```

### The Vault Connection

The run folder IS a vault. The same `sgit-ai` tooling that handles encrypted file transfer handles QA session data. Push a session to a vault. Share the vault key. Anyone with the key can review the full session.

```bash
sg-qa export abc123 --vault
# Pushes session folder to vault, prints vault key
# Recipient: sgit-ai clone <vault-key>
```

## Testing Pattern

Following the SG_Send__CLI test pattern: use `SimpleNamespace` to mock argparse args.

```python
from types    import SimpleNamespace
from unittest import TestCase

class test_CLI__QA__Run(TestCase):

    def setUp(self):
        self.cli_run = CLI__QA__Run(
            endpoint_config=Schema__QA_Endpoint_Config(
                endpoint='http://localhost:8000',
                target='local'
            )
        )

    def test__cmd_run__upload_file(self):
        args = SimpleNamespace(
            workflow='upload-file',
            target='local',
            capture='minimal',
            endpoint=None,
            api_key=None,
        )
        # cmd_run calls the FastAPI endpoint and returns result
        result = self.cli_run.cmd_run(args)
        assert result['status'] == 'pass'

    def test__cmd_list__shows_available_workflows(self):
        args = SimpleNamespace()
        workflows = self.cli_run.cmd_list(args)
        assert 'upload-file'    in workflows
        assert 'combined-link'  in workflows
```

## Three-Layer Architecture

```
Layer 1: CLI (sg-qa)
  - Triggers execution via HTTP calls to FastAPI
  - Manages sessions (folders of run artifacts)
  - Generates reports from session data
  - Exports to vault / S3
  - Computes state machine coverage from snapshots

Layer 2: FastAPI Service
  - Runs on Lambda / EC2 / local
  - Executes Playwright workflows
  - Returns results as JSON (Schema__QA_Response)
  - Records transitions_observed per workflow

Layer 3: Observability
  - Grafana (dashboards from Prometheus metrics)
  - OpenSearch (searchable logs and traces)
  - Vault (session artifacts for sharing and review)
  - QA site (state machine coverage maps, Mermaid diagrams)
```

Each layer is independent. The CLI does not know how the FastAPI works. The FastAPI does not know where its results end up. The observability layer consumes whatever data the other layers produce.
