## 1. What the Two Briefs Are Asking For

### Brief 1: `v0.16.61__dev-brief__qa-site-improvements.md` — "Enrich what's there"

The v0.16.61 brief asks for **seven content and infrastructure improvements to the existing Jekyll site**, while keeping Jekyll as the delivery mechanism:

1. **Code alongside screenshots** — show the test code that produced each screenshot, per-step
2. **Bidirectional GitHub links** — QA page → GitHub test file, and GitHub test file → QA page
3. **Commit and version provenance** — capture and display the git hash and version on every page
4. **User stories layer** — map use cases → tests → screenshots so the site is human-readable, not just a test report
5. **LLM-assisted reviews** — screenshot analysis, coverage review, documentation quality checks
6. **Release notes from tests** — auto-generate per-release pages from issue files, tests, and screenshots
7. **Static enrichment at build time** — pull git metadata, PR descriptions, change stats into the build

This brief is about **making the QA site a living product document**. It assumes Jekyll continues as the build system.

### Brief 2: `v0.17.2__brief__qa-site-migration-to-web-components.md` — "Replace the foundation"

The v0.17.2 brief supersedes v0.16.61 for delivery mechanism: instead of building enriched content into Jekyll, **replace Jekyll entirely** with the same Web Components + S3/CloudFront stack used by Send and Tools. The motivations are:

- Visual disconnect (QA site is light theme; product is Aurora dark)
- Infrastructure island (only Ruby dependency in the whole ecosystem)
- Capability ceiling (interactive components are awkward in Liquid/Jekyll)
- No shared components (header/footer already exist in Tools repo, unused here)

The v0.17.2 brief is fully designed with a side-by-side strategy: build `sg_send_qa__site_v2/` in parallel without touching the live site, verify, then cut over. It defines 5 implementation sessions plus cutover.

**Relationship between the briefs:** v0.16.61 defines *what* to build (the enriched content model). v0.17.2 defines *how* to build and deliver it (Web Components, S3/CloudFront). The v0.17.2 Web Components (`<qa-test-page>`, `<qa-screenshot-step>`) are designed to implement v0.16.61's enrichment features. They are complementary, not competing.

## 2. Assessment by Role

### Architect View

**The strategy is sound. The design is well-specified. A few gaps need addressing.**

The side-by-side migration approach (build `sg_send_qa__site_v2/` in parallel, keep Jekyll alive, cut over only after full verification) is exactly the right call. It eliminates migration risk almost entirely. The fact that v2 references screenshots via full URLs to the live site — rather than copying them — is clever: it makes the v2 directory self-contained and deployable without touching the existing pipeline.

The Web Components model (Shadow DOM, ES modules, data via JSON fetch) is the right architecture for this. It matches the rest of the SGraph ecosystem and gives us composable, testable components.

**Gaps and risks:**

1. **`v0.17.3__dev-brief__qa-pages-architectural-reviews.md` does not exist.** The v0.17.2 brief lists it as an input brief. If it contains additional requirements (architectural review process, review integration with the pipeline), those are currently invisible to the implementation team. This must be clarified or the reference removed before Session 1.

2. **The `<qa-test-page>` component has complex responsibilities.** It fetches metadata, deduplicates test methods, renders provenance, embeds source code, and wires up child components. This is the highest-risk component. The brief recommends generating syntax-highlighted HTML at build time (Pygments), which is correct — it avoids client-side JS complexity. But it creates a build-time dependency (Pygments) that must be added to `requirements.txt`.

3. **Shadow DOM and Aurora CSS variables.** The brief correctly notes that CSS custom properties from `:root` cross Shadow DOM boundaries. But `design-tokens.css` must be loaded in the top-level document, not just inside components, for this to work. The template (`test-page.html`) already does this via `<link rel="stylesheet" href="{common_prefix}css/design-tokens.css">`. This is correct and consistent.

4. **`deploy_static_site.py` assumptions.** The CI snippet in Phase 4.2 assumes `deploy_static_site.py` supports `--clean-latest`. Verify this is the case in the Send/Tools versions before copying. If the flag differs, the CI step will fail silently or noisily.

5. **The `_metadata.json` deduplication fix (§3.4)** is correct but should be validated against all 60 existing `_metadata.json` files, not just `combined_link`. The deduplication logic (`strip leading _\d+__ prefix, keep prefixed version`) may not cover all naming variants in the dataset.

**What I'd do differently:** Nothing significant. The two-generator approach (`scaffold_page()` untouched, new `scaffold_page_v2()`) is the safest pattern and correctly prevents regressions.

### Developer View

**The implementation scope is clearly defined. The hardest part is `<qa-test-page>`. Recommended order: CSS → sidebar → screenshot-step → test-page → generator.**

**What needs to be built (in order):**

| # | Artefact | Where | Size | Notes |
|---|----------|-------|------|-------|
| 1 | `sg_send_qa__site_v2/` directory scaffold | repo | trivial | mkdir + copy assets |
| 2 | `qa-site.css` | `_common/css/` | small | layout CSS, already spec'd in brief |
| 3 | `qa-code.css` | `_common/css/` | small | generate from Pygments `monokai` style, customise to Aurora variables |
| 4 | `qa__run-locally.sh` | `scripts/` | small | verbatim from brief, already spec'd |
| 5 | `<qa-sidebar>` | `_common/js/components/` | medium | fetch + render JSON, current-page highlight, group toggle |
| 6 | `<qa-screenshot-step>` | `_common/js/components/` | medium | card layout, deterministic toggle, code context |
| 7 | `<qa-test-page>` | `_common/js/components/` | large | all sections, metadata fetch, dedup, provenance |
| 8 | `<qa-dashboard>` | `_common/js/components/` | medium | fetch `qa_summary.json`, render coverage table |
| 9 | HTML page template | `_common/templates/` | small | placeholder substitution, already spec'd |
| 10 | Static pages (homepage, pipeline, etc.) | `pages/` | medium | 6 pages, hand-crafted HTML |
| 11 | `scaffold_page_v2()` + `generate_v2()` | `QA_Generate_Docs.py` | medium | new methods, don't touch existing |

**Hardest part:** `<qa-test-page>`. It has the most moving parts: metadata fetch, deduplication logic, provenance rendering, syntax-highlighted source code embedding. The brief's recommendation to embed metadata as `<script type="application/json" id="qa-metadata">` at generation time (avoiding a second fetch at runtime) is the right call — do it this way.

**What can be reused:** The `QA_Generate_Docs` class already walks the use-case directory structure, reads `_metadata.json`, and writes output files. The `generate_v2()` method is the same walk with a different output format. Reuse the directory traversal logic directly.

**Right order of implementation:** CSS and fonts first (visual foundation). Then sidebar (navigation context). Then screenshot-step (isolated, no dependencies). Then test-page (composes the above). Then dashboard. Then generator. This order allows visual verification at each step before the next component builds on it.

**Concrete concern:** The brief says to read files from the Send and Tools repos as prerequisites (Phase 0 list). Those repos are not in this working tree. The implementation sessions need to either clone them to `/tmp/` first or have those assets pre-copied into `sg_send_qa__site_v2/_common/` before Session 1 begins. The local dev script (`qa__run-locally.sh`) cannot run until the CSS and font files are actually present.

### DevOps View

**The CI strategy is correct. Three things need to happen before Session 4 (the cutover): AWS credentials in repo secrets, CloudFront/Route53 infra created, and `deploy_static_site.py` copied into the repo.**

**What changes to CI/CD are needed:**

| Phase | CI Change | When |
|-------|-----------|------|
| Sessions 1–3 | None — existing Jekyll pipeline continues unchanged | Throughout side-by-side |
| Session 3 | Add `generate_docs_v2` step to CI (runs in parallel with existing, writes to `sg_send_qa__site_v2/`) | After v2 generator is built |
| Session 4 | Add S3 deploy step for v2 site (deploy to `qa-v2.send.sgraph.ai`) | After Sessions 1–3 verified |
| Cutover | Remove Jekyll steps, replace with S3 deploy to `qa.send.sgraph.ai` | After `qa-v2` approved |
| Post-cutover | Delete Jekyll artifacts (Gemfile, `_config.yml`, `_layouts/`) | Very last step |

**How S3/CloudFront replaces GitHub Pages:**

GitHub Pages serves from the `gh-pages` branch (or `docs/`), with a CNAME record pointing `qa.send.sgraph.ai` to `the-cyber-boardroom.github.io`. The cutover is a Route53 record change:

```
Before: qa.send.sgraph.ai CNAME → the-cyber-boardroom.github.io
After:  qa.send.sgraph.ai CNAME → dXXXXXX.cloudfront.net
```

The Jekyll build and `deploy-pages@v4` action steps are deleted. The new `deploy_static_site.py` S3 upload runs instead. CloudFront serves the content.

**Side-by-side means two concurrent deploy targets in CI:**

During Sessions 3–4, the CI pipeline runs two deploy paths:
1. Existing: Jekyll build → `deploy-pages@v4` → `qa.send.sgraph.ai` (unchanged)
2. New: `generate_docs_v2` → `deploy_static_site.py` → S3 → `qa-v2.send.sgraph.ai` (new)

Both run on every push. This costs essentially nothing (S3 uploads are cheap; CloudFront serves from cache). The CI time increase is minimal.

**Pre-Session-4 blockers (things that require Dinis/human action):**

| Blocker | Who | What |
|---------|-----|------|
| Create S3 prefix `websites/sgraph-qa-send/` | Dinis | One-time setup in existing bucket |
| Create CloudFront distribution for `qa-v2.send.sgraph.ai` | Dinis | New distribution, origin path `/websites/sgraph-qa-send/dev/latest` |
| ACM certificate for `qa-v2.send.sgraph.ai` | Dinis | In `us-east-1` (CloudFront requirement) |
| Route53 CNAME for `qa-v2.send.sgraph.ai` | Dinis | Points to new CloudFront distribution |
| Add CI secrets to QA repo | Dinis | `QA_S3_BUCKET`, `QA_CLOUDFRONT_DIST`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` |

None of these are blockers for Sessions 1–3. They only block Session 4.

**Pipeline risk for the parallel-generator phase:** The v2 generator writes to `sg_send_qa__site_v2/`. If CI commits this directory back to the repo (as it does with screenshots in the existing pipeline), the repo will grow quickly — 60 HTML pages per CI run. Recommendation: do NOT commit the v2 site directory to git. Generate it at CI time from source, deploy to S3, and `.gitignore` the `sg_send_qa__site_v2/pages/use-cases/` generated output. Only the static assets (`_common/`, templates, hand-crafted pages) live in git.

## 3. Proposed Approach — How to Actually Do This

**Four phases. Sessions 1–3 are zero-risk. Session 4 requires Dinis to provision infra first.**

### Phase A: Foundation (Session 1) — "The shell works"

**Goal:** A locally serveable v2 site with Aurora styling and two real pages. Existing site untouched.

1. Clone Send and Tools repos to `/tmp/` (read-only reference)
2. Create `sg_send_qa__site_v2/` directory structure (as per brief)
3. Copy `design-tokens.css`, `shared-components.css`, `fonts.css`, `*.woff2`, `favicon.svg` from Send v0.3.0 into `sg_send_qa__site_v2/_common/`
4. Write `qa-site.css` (verbatim from brief — layout CSS is already specified)
5. Generate `qa-code.css` from Pygments once, commit the static file
6. Write `qa__run-locally.sh` (verbatim from brief)
7. Copy `_data/qa_sidebar.json` and `_data/qa_summary.json` into `sg_send_qa__site_v2/_data/`
8. Write `pages/index.html` (homepage, Aurora theme, no Web Components needed yet — just static HTML)

**Verify:** `./scripts/qa__run-locally.sh` → browse to `http://localhost:10064/pages/` → Aurora dark theme renders → fonts load → layout correct. Jekyll site untouched on port 4000 (or just confirmed via `git status` — no changes to `sg_send_qa__site/`).

### Phase B: Web Components (Session 2) — "The components work"

**Goal:** `<qa-sidebar>`, `<qa-screenshot-step>`, `<qa-test-page>`, `<qa-dashboard>` all render real data.

Build in this order (each depends on the previous):
1. `qa-sidebar.js` — reads `_data/qa_sidebar.json`, renders navigation
2. `qa-screenshot-step.js` — renders a single screenshot card (test in a standalone fixture page)

4. `qa-dashboard.js` — reads `_data/qa_summary.json`
5. Wire `<qa-sidebar>` into the homepage and pipeline pages
6. Create one hand-crafted `pages/use-cases/02-upload-share/combined_link/index.html` with hardcoded metadata to validate `<qa-test-page>` rendering before the generator exists

**Verify:** Load `combined_link/index.html` locally — all sections render, screenshots load from `https://qa.send.sgraph.ai/...`, sidebar highlights current page, deterministic toggle works.

### Phase C: Generator (Session 3) — "All 60 pages work"

**Goal:** `generate_docs_v2` produces valid HTML for all 60 existing pages. No Jekyll changes.

1. Add `scaffold_page_v2()` method to `QA_Generate_Docs` (new method, don't touch `scaffold_page()`)
2. Add `generate_v2()` entry point
3. Add `_deduplicate_tests()` — validate against all 60 `_metadata.json` files
4. Add Pygments syntax highlighting at build time; `qa-code.css` already committed
5. Add `generate_docs_v2` CLI entry point
6. Run `python -m sg_send_qa.cli.generate_docs_v2` — verify all 60 pages generate without errors
7. Convert remaining hand-crafted pages: dashboard, roadmap, known-bugs, change-requests
8. Add the v2 generator to CI (generates but does NOT deploy yet)
9. Run full 10-point Verification Checklist from the brief

**Verify:** All 60 use-case pages + 6 static pages render correctly at `http://localhost:10064/`. Sidebar navigation works across all pages. Screenshot images load. `git status` shows `sg_send_qa__site/` is completely unchanged.

### Phase D: Deploy + Cutover (Session 4 — requires Dinis infra first)

**Prerequisite check:** Dinis has confirmed all 5 infra items from the DevOps section above are in place (S3, CloudFront, Route53, ACM, CI secrets).

1. Copy `deploy_static_site.py` from Send repo into `scripts/`
2. Add `deploy-qa-site-v2` CI job (deploys to S3/CloudFront staging at `qa-v2.send.sgraph.ai`)
3. Push — verify the full CI pipeline passes and `https://qa-v2.send.sgraph.ai` is live
4. Share `qa-v2` URL with Dinis for approval

## 5. Concerns and Pushback

**None that block execution.** The brief is exceptionally detailed and well-designed. The implementation order is correct, the risk model is sound, and the side-by-side strategy eliminates the main failure mode (breaking the live site).

**Minor concerns:**

1. **Scope of `<qa-test-page>` vs `scaffold_page_v2()`.** The brief proposes embedding syntax-highlighted HTML and metadata JSON into the HTML page at generation time (server-side, via Pygments). This is the right call for performance and offline use. However, it means the component relies on this embedded data being present — it should gracefully handle the case where the inline data is missing (fall back to fetching `_metadata.json` via HTTP). This makes the component more robust and useful as a standalone element.

2. **The `v0.16.61` improvements (GitHub links, provenance, user stories, LLM reviews, release notes) are not all addressable in the v0.17.2 sessions.** The v0.17.2 brief addresses #1 (code + screenshots), #3 (provenance), and #4 (user story via `module_doc`). Items #2 (bidirectional GitHub links), #5 (LLM reviews), #6 (release notes), and #7 (static enrichment) are future work. The team should make sure Dinis knows which v0.16.61 items are in scope for this migration vs deferred. Based on reading both briefs, our understanding is that the migration covers items 1, 3, and 4 of v0.16.61; the rest are post-cutover.

3. **`qa-code.css` must be generated once and committed.** It is not generated at build time on every CI run. The brief implies this (`Do this once, commit the CSS file`). This is correct — but make sure this is done in Session 1 (foundation), not Session 3 (generator), because the template references it from the start.

4. **The `_metadata.json` deduplication fix removes old-style method names.** Before removing them from the rendered output, verify the old-style names are not referenced by any other system (e.g., CI dashboard, issue trackers, or QA summary JSON). If `qa_summary.json` uses old-style names as keys, deduplication could break the dashboard component.

## 6. Resolved Answers (from Dinis, 26 March 2026)

The following open questions from Section 4 have been answered:

### Q1 — Missing `v0.17.3` brief
**Answer:** Ignore the reference. It is not a prerequisite for this migration. Not a blocker.

### Q2 — Should v2 site be committed to git or generated fresh in CI?
**Answer:** **Commit to git, then use those committed files to push to S3.**

Rationale: git becomes the historical record of every version of the site. Enables future optimisation where the S3 deploy only pushes changed files (by diffing the previous commit). A full site rebuild remains an option, but the diff path is available. This means `sg_send_qa__site_v2/` (including generated pages) is committed on every CI run that regenerates documentation — same pattern as the existing `sg_send_qa__site/` directory.

**Implementation impact:**
- Do NOT `.gitignore` the generated `pages/use-cases/` output directory inside `sg_send_qa__site_v2/`
- The CI pipeline must commit the generated v2 HTML pages back to the branch (alongside the existing screenshot auto-commit step)
- The S3 deploy reads from the committed files, not from a build artefact

### Q3 — Should the v2 site use shared `sg-site-header` and `sg-site-footer` from the Tools repo?
**Answer:** **Yes.** Use the versions from the Tools repo, available at `https://dev.tools.sgraph.ai/` and `https://sgraph.ai`.

**Implementation impact:**
- The `<sg-site-header>` and `<sg-site-footer>` Web Components must be loaded on every page
- These are CDN-hosted from the Tools site — do not vendor/copy them (use the CDN `<script type="module">` import pattern, same as the Tools pages)
- The `<qa-sidebar>` component is QA-specific and still needed for the test navigation sidebar, but the global header and footer come from the shared components
- Page template (`_common/templates/test-page.html`) must include both the shared header/footer script imports AND the sidebar script import
- This resolves the "visual disconnect" problem more completely — the QA site will share the exact same header/footer chrome as `send.sgraph.ai` and `tools.sgraph.ai`
- Check `https://dev.tools.sgraph.ai/` for the current component CDN URLs before Session 1

### Q4 — Target version string for the v2 site
**Answer:** The v2 site is a new IFD (independent deployable artefact) — it starts at **`v0.1.0`**.

**Implementation impact:**
- The v2 site has its own version file or version constant, separate from the `sg_send_qa` Python package version
- `deploy_static_site.py` is called with `--version v0.1.0` (or read from a dedicated `sg_send_qa__site_v2/version` file)
- The existing `sg_send_qa/version` file (currently `v0.1.0` for the Python package) and the v2 site version are independent — they track different things (the test runner package vs the documentation site)
- Recommended: create `sg_send_qa__site_v2/version` containing `v0.1.0` as the site's own version file, matching the IFD pattern used by the Send UI (`v0/v0.1/v0.1.0/`)

### Q5 — GitHub URL for test source links
**Answer:** The stable repo URL is `https://github.com/the-cyber-boardroom/SG_Send__qa`.

**Implementation impact:**
- `QA_Generate_Docs.scaffold_page_v2()` constructs GitHub links as:
  `https://github.com/the-cyber-boardroom/SG_Send__qa/blob/{commit_sha}/{relative_test_path}`
- The `data-github-url` attribute on `<qa-test-page>` receives this full URL
- The commit SHA comes from `git rev-parse HEAD` at generation time (available in CI as `$GITHUB_SHA`)
