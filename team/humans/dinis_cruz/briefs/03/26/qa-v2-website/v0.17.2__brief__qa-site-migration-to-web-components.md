## Why This Migration

The QA site at qa.send.sgraph.ai is currently a **Jekyll static site** (Ruby, Liquid templates, GitHub Pages). The main product (send.sgraph.ai) and the tools site (tools.sgraph.ai) both use **vanilla HTML + Web Components**, deployed via `deploy_static_site.py` to **S3 + CloudFront**.

This creates four problems:

1. **Visual disconnect.** The QA site uses a generic light theme (blue accent, white background, system fonts). The product uses the Aurora dark theme (teal accent, navy background, DM Sans + JetBrains Mono). The screenshots show Aurora; the documentation wrapper doesn't. The site looks like it belongs to a different project.

2. **Infrastructure island.** The QA site requires Ruby 3.3, Bundler, and Jekyll gems вҖ” dependencies that appear nowhere else in the SGraph ecosystem. Both other sites use zero Ruby. The deploy pipeline is GitHub Pages (separate from the S3/CloudFront infrastructure that everything else uses).

3. **Capability ceiling.** The enriched QA pages described in the v0.16.61 and v0.17.3 briefs need interactive elements: collapsible code with syntax highlighting, code-to-screenshot linking, blast radius visualisation, embedded video. These are natural in Web Components and awkward in Jekyll markdown.

4. **No shared components.** `sg-site-header`, `sg-site-footer`, `sg-locale-picker` are already built and CDN-hosted from the Tools repo. The QA site doesn't use them. It has its own sidebar and layout in a monolithic `default.html`.

The infrastructure to do this properly already exists. The migration is about **using what we've already built** rather than maintaining a parallel stack.

## Architecture Decision

**Replace Jekyll with the same static HTML + Web Components model used by Send and Tools.**

- Pages become standalone HTML files (not markdown compiled by Jekyll)
- Layout uses Web Components: `sg-site-header`, `sg-site-footer`, new `<qa-sidebar>`
- Design uses Aurora theme via `design-tokens.css`
- `QA_Generate_Docs` outputs HTML directly (not markdown with front matter)
- Deploy via `deploy_static_site.py` to S3 + CloudFront
- Local dev via `qa__run-locally.sh` with `python3 -m http.server`
- Zero Ruby dependency

## Side-by-Side Strategy: Build New, Cut Over When Ready

**Critical principle: the existing Jekyll site stays untouched until the new site is fully working.**

The migration builds a complete new site in a parallel directory. The existing `sg_send_qa__site/` continues to serve qa.send.sgraph.ai via Jekyll + GitHub Pages throughout. No files are deleted, no markdown is modified, no CI pipeline is changed вҖ” until the new site has been verified and the team decides to cut over.

The key insight: **the v2 site is completely standalone**. It contains no screenshots locally. Instead, the generated HTML pages reference screenshots via full URLs to the live site:

```html
<!-- In the v2 generated page for combined_link -->
<img src="https://qa.send.sgraph.ai/pages/use-cases/02-upload-share/combined_link/screenshots/01_file_selected.png"
     alt="01 File Selected">
```

This means:
- **No symlinks.** No cross-directory references. No file sharing. The v2 directory is self-contained and deployable as-is.
- **No duplication.** Screenshots are not copied вҖ” they're loaded from the existing live site.
- **Clean deployment.** When `sg_send_qa__site_v2/` is deployed to S3, it contains only HTML, CSS, JS, and fonts. Screenshots are served from the existing `qa.send.sgraph.ai` until the tests are updated.
- **Gradual migration.** As tests are updated to write screenshots into the v2 site structure, those images become local and the full URLs are replaced with relative paths. This happens naturally, one test at a time, not as a big-bang migration.

### The Two-Generator Approach

The safest path for `QA_Generate_Docs` is to add a parallel method:

```python
class QA_Generate_Docs(Type_Safe):
    site_dir      : str = "sg_send_qa__site"        # existing Jekyll site
    site_v2_dir   : str = "sg_send_qa__site_v2"     # new Web Components site

    # Base URL for screenshots on the live site (used by v2 generator)
    live_site_url : str = "https://qa.send.sgraph.ai"

    def scaffold_page(self, ...):
        """Existing method вҖ” generates markdown for Jekyll. UNCHANGED."""
        ...

    def scaffold_page_v2(self, ...):
        """New method вҖ” generates HTML for Web Components site.
        Screenshot URLs point to self.live_site_url."""
        ...

    def generate(self):
        """Existing вҖ” generates into site_dir. UNCHANGED."""
        ...

    def generate_v2(self):
        """New вҖ” generates into site_v2_dir using scaffold_page_v2."""
        ...
```

During development, run both: `python -m sg_send_qa.cli.generate_docs` (existing) AND `python -m sg_send_qa.cli.generate_docs_v2` (new). Both read the same `_metadata.json` files from the existing site. The v2 generator embeds full URLs for screenshots. Compare the sites page by page.

### Screenshot URL Strategy

**Phase 1 (during development):** All screenshot `<img>` tags use full URLs:
```html
<img src="https://qa.send.sgraph.ai/pages/use-cases/02-upload-share/combined_link/screenshots/01_file_selected.png">
```

**Phase 2 (after cutover):** As tests are updated to write screenshots into the v2 site, the URLs switch to relative paths:
```html
<img src="screenshots/01_file_selected.png">
```

The `<qa-screenshot-step>` component should accept both forms вҖ” a full URL or a relative path. The `QA_Generate_Docs` v2 generator should have a flag (`--local-screenshots`) that switches between full URLs and relative paths. Default is full URLs during the migration period.

### Verification Checklist (Before Cutover)

Before any changes to the live site, ALL of the following must pass:

| # | Check | How |
|---|---|---|
| 1 | Every use-case page renders correctly in v2 | `qa__run-locally.sh` вҶ’ browse all 60 pages |
| 2 | Screenshot images load on every page | Visual scan or automated link checker |
| 3 | Sidebar navigation works (current page highlighted, all links resolve) | Click through all groups |
| 4 | Dashboard renders with real data | Load dashboard, compare numbers to Jekyll version |
| 5 | Mobile layout works | Resize to 768px, verify sidebar collapse |
| 6 | Print layout works | Print preview 2-3 pages |
| 7 | No broken links | `wget --spider --recursive` on local server |
| 8 | Page count matches: v2 has same number of pages as Jekyll | `find . -name "index.html" | wc -l` vs `find . -name "*.md" | wc -l` |
| 9 | `qa__run-locally.sh` starts in <2 seconds (no build step) | Time it |
| 10 | v2 generator produces identical page list to v1 generator | Compare output logs |

### Cutover Sequence (Only After Verification)

Once all checks pass:

1. **Create `qa-v2.send.sgraph.ai` infrastructure** вҖ” CloudFront distribution + Route53 CNAME + ACM cert (see Phase 4.4)
2. **Deploy v2 to staging** вҖ” `deploy_static_site.py` pushes to S3, CloudFront serves at `qa-v2.send.sgraph.ai`
3. **Verify on staging** вҖ” run full verification checklist against `https://qa-v2.send.sgraph.ai`
4. **Share staging URL with team** вҖ” Dinis and anyone else can compare `qa-v2` vs `qa` side-by-side in the browser
5. **Get approval** вҖ” explicit go from Dinis before touching the live site
6. **Switch `qa.send.sgraph.ai`** вҖ” repoint Route53 to CloudFront (see Phase 4.5)
7. **Monitor for 24h** вҖ” watch for broken links, missing images
8. **Clean up** вҖ” delete Gemfile, `_config.yml`, `_layouts/`, old `.md` page files, `scaffold_page()` v1 method

If anything goes wrong at steps 6-7: revert Route53 to GitHub Pages CNAME. The Jekyll site is still intact and deployable. `qa-v2.send.sgraph.ai` continues to work independently regardless.

Each CCW session produces working, testable output WITHOUT touching the live site:

### Phase 1: Create the QA-Specific Web Components

Build three new Web Components in `sg_send_qa__site_v2/_common/js/components/`:

#### 1.1 `<qa-sidebar>` Component

**File:** `sg_send_qa__site_v2/_common/js/components/qa-sidebar.js`

Reads `qa_sidebar.json` (fetch from relative path), renders the grouped navigation sidebar. Replaces the Jekyll Liquid sidebar from `default.html`.

Requirements:
- Shadow DOM for style encapsulation
- Uses Aurora CSS variables from `design-tokens.css` (which cross Shadow DOM boundaries via `:root`)
- Highlights current page based on `window.location.pathname`
- Groups with emoji icons (from `qa_sidebar.json`)
- вҳ… markers for composite tests (from `qa_sidebar.json` `composite` field)
- Static navigation links at top: Home, Dashboard, Pipeline, Roadmap, Known Bugs
- External links at bottom: SG/Send App, GitHub Repo
- Responsive: sidebar becomes top nav on mobile (matches current `@media (max-width: 768px)` behaviour)
- Collapsible groups (click group heading to toggle)

Data source: `/_data/qa_sidebar.json` (already generated by `QA_Generate_Docs.write_sidebar_data()`)

#### 1.2 `<qa-test-page>` Component

**File:** `sg_send_qa__site_v2/_common/js/components/qa-test-page.js`

The enriched test page вҖ” implements the vision from both briefs. Fetches its own metadata and renders the full page content.

Attributes:
- `data-use-case` вҖ” use case name (e.g. `combined_link`)
- `data-group` вҖ” group id (e.g. `02-upload-share`)
- `data-metadata-url` вҖ” relative path to `_metadata.json` (default: `screenshots/_metadata.json`)

Sections rendered (in order):
1. **Title** вҖ” from use case name, title-cased
2. **Provenance line** вҖ” commit hash, version, timestamp (from metadata or page data attribute)
3. **User story** вҖ” from `module_doc` in metadata. Rendered as a blockquote callout.
4. **Test methods table** вҖ” method name, description, screenshot count (from `tests` array in metadata). Filter duplicates (the metadata currently has old-style and new-style method names вҖ” deduplicate by stripping leading `_NN__` prefixes and matching).
5. **Screenshot gallery** вҖ” each screenshot rendered as a `<qa-screenshot-step>` (see 1.3)
6. **Test source code** вҖ” full test code, syntax highlighted, in a collapsible `<details>` block. Code comes from a `data-source-url` attribute pointing to the raw test file, or embedded inline via a `<script type="text/qa-source">` block in the page HTML.
7. **GitHub link** вҖ” link to test source on GitHub

Syntax highlighting: Use a lightweight client-side highlighter. Options (in preference order):
- Prism.js (lightweight, widely used, supports Python) вҖ” can be vendored
- Highlight.js вҖ” heavier but also works
- Or: generate pre-highlighted HTML at build time in `QA_Generate_Docs` using `pygments` (Python, already available in the test environment)

**Recommended approach:** Generate syntax-highlighted HTML in `QA_Generate_Docs` using `pygments` at page generation time. This means zero client-side JS for highlighting, and the output is static HTML with CSS classes. Add a `qa-code.css` file with Pygments styles in Aurora theme colours.

#### 1.3 `<qa-screenshot-step>` Component

**File:** `sg_send_qa__site_v2/_common/js/components/qa-screenshot-step.js`

Renders a single screenshot with its context. Used inside `<qa-test-page>`.

Attributes:
- `name` вҖ” screenshot name (e.g. `01_file_selected`)
- `description` вҖ” text description
- `screenshot-url` вҖ” path to the PNG
- `deterministic-url` вҖ” path to the deterministic PNG (optional)
- `code-context` вҖ” the code line(s) that triggered this screenshot (optional вҖ” for v0.17.3 enrichment)

Layout: Card-style container with:
- Step number and title (derived from name: `01_file_selected` вҶ’ "01 В· File Selected")
- Description text
- Screenshot image (full width within the card, bordered, with subtle shadow)
- Deterministic toggle вҖ” "Show deterministic view" as a toggle switch, not a collapsed `<details>` (the current approach confuses non-QA users)
- Code context вҖ” if present, a small code block above the screenshot showing the triggering lines

Design notes:
- Card background: `var(--color-surface)` (#1E2A4A)
- Border: `var(--color-border)` with subtle teal
- Screenshot border-radius: 6px
- Step number: `var(--color-primary)` (#4ECDC4) badge

## Key Constraints

1. **The screenshot directory structure must not change.** Tests write screenshots to `sg_send_qa__site/pages/use-cases/{group}/{use_case}/screenshots/`. This path is hardcoded in the test harness. The migration changes the documentation page format, not the screenshot storage.

2. **`_metadata.json` format is unchanged.** The Web Components read the existing JSON schema. No migration of metadata files needed.

3. **`qa_sidebar.json` and `qa_summary.json` formats are unchanged.** Same generators, same output, new consumers (Web Components instead of Liquid templates).

4. **No external CDN dependencies at test time.** The QA site must work from a local `python3 -m http.server` with no network. All assets (fonts, CSS, JS) must be vendored locally.

5. **Follow IFD patterns.** Web Components use Shadow DOM. CSS custom properties from `design-tokens.css` cross Shadow DOM boundaries via `:root`. No CSS-in-JS, no build tools, no npm.
