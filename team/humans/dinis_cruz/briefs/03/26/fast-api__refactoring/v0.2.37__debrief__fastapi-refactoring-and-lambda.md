## Executive Summary

The two briefs are a natural pair. Brief v0.17.2 cleans up the QA API internals (replace vanilla FastAPI with `Serverless__Fast_API`, merge two apps into one, adopt `Fast_API__Routes`). Brief v0.16.61 specifies how that API gets deployed serverlessly on Lambda with hot-swap code from S3.

**The dependency is clear: do the FastAPI refactor first.** A `Serverless__Fast_API` subclass has the `.handler()` method that Mangum needs for Lambda. Until the refactor is done, the Lambda work has no clean entry point.

**Overall assessment:** Both briefs are well-specified and the current codebase is closer to the target than it might look. The foundation is solid (`Type_Safe` schemas, clean session/runner separation, `osbot_fast_api_serverless` already in requirements). The refactor is mostly mechanical with a few design decisions. The Lambda work is a container/DevOps task once the refactor is done.

## Architect Perspective

### What brief v0.17.2 is asking for

The brief asks us to stop using vanilla FastAPI primitives and adopt the osbot patterns already used in the SG/Send main repo. The target state:

- `Fast_API__QA__Server` extends `Serverless__Fast_API`
- One unified server (merge `QA_API__Server.py` + `server/main.py`)
- Route modules become `Fast_API__Routes` subclasses
- Service injection via Type_Safe constructor kwargs, not `Depends()`
- Body params typed directly as `Schema__QA_Request` (no `dict = Body({})`)

### Current vs target gap

| Component | Current | Target | Effort |
|-----------|---------|--------|--------|
| App class | `FastAPI()` instance in `QA_API__Server.py` | `Serverless__Fast_API` subclass | Medium — lifecycle methods to wire |
| Second app | `server/main.py` (separate FastAPI, test runner + docs) | Merged into same subclass | Small — only if `server/main.py` exists; currently absent as a file |
| Route classes | `APIRouter` + module-level functions | `Fast_API__Routes` subclasses | Medium — 4 route files × ~3 endpoints each |
| Body params | `dict = Body(default={})` + `_req()` helper | `Schema__QA_Request` body param | Small — straightforward swap |
| Service injection | `QA_API__Runner(request=req)` instantiated per route call | Runner injected at startup via `add_routes()` | Medium — requires runner to be request-agnostic |
| Test infra | No singleton test client | `Fast_API__Test_Objs__QA` singleton | Small |
| Version string | Hardcoded `"0.2.30"` in `QA_API__Server.py` and health endpoint | Read from `sg_send_qa/version` | Trivial |

### Key design decision: runner injection vs per-request instantiation

Currently every route does `QA_API__Runner(request=req).run(workflow)` — the runner is stateless and recreated per request. The target pattern injects the runner at startup.

**Issue:** `QA_API__Runner` currently holds the `request` as an attribute. For it to be injected at the app level, `request` must be passed to `.run()` rather than stored on the runner.

**Recommendation:** Rename to `run_with_request(request, workflow_fn)` and make `QA_API__Runner` stateless at the class level. This is a small change but a prerequisite for proper injection.

### The two-app merge

`server/main.py` (test runner + docs browser) doesn't exist as a file right now. The CLAUDE.md lists it in the repo structure but it's absent. This means the merge is actually free — there is nothing to merge yet. The new `Fast_API__QA__Server` becomes the single server from the start.

### Lambda compatibility

`Serverless__Fast_API` inherits from `Mangum` via the `.handler()` method. Once the refactor is done, the Lambda entry point is:

```python
with Fast_API__QA__Server() as server:
    server.setup()
    handler = server.handler()   # Mangum-wrapped ASGI app
    app     = server.app()       # for local dev / Lambda Web Adapter

def run(event, context=None):
    return handler(event, context)
```

This is exactly what brief v0.16.61 expects as the `HANDLER`. Clean.

## Developer Perspective

### Assessment of existing code quality

The existing code is clean and well-structured. The main patterns are already right:
- `Type_Safe` throughout (schemas, runner, session)
- Session/runner separation is solid — `QA_API__Session` handles browser lifecycle, `QA_API__Runner` handles error/timing/trace
- No mocks, real browser sessions
- `__enter__`/`__exit__` on session = Lambda-friendly context manager

The things the brief calls out as wrong (`dict = Body({})`, `APIRouter`, hardcoded version) are real but they're cosmetic. The logic underneath is sound.

### Refactoring steps (concrete)

**Step 1: Fix `QA_API__Runner` to be injection-friendly**
```python
# Change run() signature so request is passed in, not stored on self
def run(self, request: Schema__QA_Request, workflow_fn) -> dict:
    ...
```
This unblocks injection at startup.

**Step 2: Create `Fast_API__QA__Server`**
```python
from osbot_fast_api_serverless.fast_api.Serverless__Fast_API import Serverless__Fast_API
from sg_send_qa.utils.Version import version__sg_send_qa

class Fast_API__QA__Server(Serverless__Fast_API):
    qa_runner: QA_API__Runner = None

    def setup(self):
        with self.config as _:
            _.name        = 'SG/Send QA API'
            _.version     = version__sg_send_qa
            _.description = 'Browser automation API for SG/Send QA'
            _.enable_api_key = False
        if self.qa_runner is None:
            self.qa_runner = QA_API__Runner()
        return super().setup()

    def setup_routes(self):
        self.add_routes(Routes__Info)
        self.add_routes(Routes__Browser,  qa_runner=self.qa_runner)
        self.add_routes(Routes__Upload,   qa_runner=self.qa_runner)
        self.add_routes(Routes__Download, qa_runner=self.qa_runner)
        self.add_routes(Routes__Workflow, qa_runner=self.qa_runner)
```

**Step 3: Convert route files**
Replace each `APIRouter` module with a `Fast_API__Routes` subclass. For example `routes__browser.py`:
```python
class Routes__Browser(Fast_API__Routes):
    tag       : str           = 'browser'
    qa_runner : QA_API__Runner

    def open(self, body: Schema__QA_Request) -> dict:
        ...
        return self.qa_runner.run(body, workflow)

    def screenshot(self, body: Schema__QA_Request) -> dict:
        ...

    def setup_routes(self):
        self.add_route_post(self.open)
        self.add_route_post(self.screenshot)
        return self
```

Note: `upload` routes have extra body fields (`filename`, `content`) beyond `Schema__QA_Request`. These should be captured in `body.options` or a new `Schema__Upload_Request` that extends `Schema__QA_Request`.

**Step 4: Update test infrastructure**
Create `Fast_API__Test_Objs__QA` singleton and update any test files that hit the API directly to use `fast_api.client()`.

### What doesn't need to change

- `QA_API__Session` — already Lambda-compatible, no changes needed
- `Schema__QA_Request`, `Schema__QA_Response`, `Schema__Capture_Config` — already correct Type_Safe schemas
- All the browser page objects and workflow helpers — untouched by this refactor

### Risk: `upload` route extra params

The `routes__upload.py` routes pull `filename` and `content` directly from the raw `body` dict. After the refactor, these need a home. Options:
1. Add them to `Schema__QA_Request.options` dict (simplest, slightly looser)
2. Create `Schema__Upload_Request(Schema__QA_Request)` with `filename: str` and `content: str`

**Recommendation:** Option 2 — typed schemas are better. One new file, minimal change.

## DevOps Perspective

### Brief v0.16.61 assessment

The architecture is sound and proven (the brief notes it's already deployed for OS-bot). The hot-swap S3 zip pattern solves the right problem: Playwright/Chromium cold-start cost is already high (~3-5s) — we don't want to add a container rebuild cycle to every code change.

### Dependency on FastAPI refactor

The Lambda entry point needs a clean `handler` function. Once `Fast_API__QA__Server.handler()` exists via `Serverless__Fast_API`, the boot loader can simply:

```python
# boot_loader.py (inside the container)
import sys, os, boto3, zipfile, tempfile

s3        = boto3.client('s3')
zip_path  = os.environ['CODE_ZIP_S3_PATH']  # e.g. s3://bucket/qa-code/latest.zip
handler_  = os.environ.get('HANDLER', 'main.run')

# Download + extract
tmp = tempfile.mkdtemp()
s3.download_file(bucket, key, f'{tmp}/code.zip')
with zipfile.ZipFile(f'{tmp}/code.zip') as z:
    z.extractall(f'{tmp}/code')
sys.path.insert(0, f'{tmp}/code')

# Import and expose handler
module_name, fn_name = handler_.rsplit('.', 1)
import importlib
module  = importlib.import_module(module_name)
handler = getattr(module, fn_name)
```

### Container contents

| Component | Source | Notes |
|-----------|--------|-------|
| Python 3.12 | `python:3.12-slim` | Base image |
| Playwright + Chromium | `pip install playwright && playwright install chromium` | ~300MB, the expensive part |
| boto3 | pip | S3 access |
| Boot loader | `boot_loader.py` in container | Only file that needs to be in the container |
| Lambda Web Adapter | AWS extension layer | Enable via env var |
| `osbot_fast_api_serverless` | pip | Needed at container level for import |

**Container size estimate:** ~800MB-1.2GB. This is within Lambda container limits (10GB) but cold starts will be 5-15s. Accept this — it's the cost of headless Chromium.

### Two-mode support

The brief asks for both direct invoke (scheduled runs) and HTTP mode (Lambda Web Adapter for on-demand API calls). The simplest implementation:

```python
# main.py (in the S3 zip)
from sg_send_qa.api.Fast_API__QA__Server import Fast_API__QA__Server

fast_api = Fast_API__QA__Server().setup()
handler  = fast_api.handler()   # Mangum handler (direct invoke)
app      = fast_api.app()       # ASGI app (Lambda Web Adapter HTTP mode)

def run(event, context=None):
    return handler(event, context)
```

Lambda Web Adapter wraps `app` automatically when `AWS_LAMBDA_RUNTIME_API` is present and the extension is configured. No code branch needed.

### CI pipeline assessment

Current CI pipeline is solid. The Lambda work adds a new deployment step but doesn't change the test pipeline. What will need adding:

1. **ECR push workflow** — triggered on Dockerfile change, pushes new container image
2. **S3 zip deploy step** — on merge to `main`, zip `sg_send_qa/` and upload to S3 (`latest.zip` + versioned copy)
3. **Lambda smoke test** — after S3 upload, invoke Lambda directly and assert pass

These are additive — the existing CI pipeline continues to run unchanged.

### Open questions for DevOps

1. **Which AWS account?** The brief assumes ECR, S3, Lambda exist. Do we have an AWS account configured? What are the bucket names?
2. **Lambda Web Adapter version** — need to verify the AWS extension ARN for the Lambda Web Adapter in the target region.
3. **IAM role** — Lambda execution role needs `s3:GetObject` on the code bucket.
4. **GitHub Actions secrets** — `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` (or OIDC) needed for S3 upload step.
