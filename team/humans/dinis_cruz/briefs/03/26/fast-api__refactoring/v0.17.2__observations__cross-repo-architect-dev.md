## 2. Cross-Repo Coupling Is Clean But Undocumented

The QA repo imports directly from the main repo's installed package:

```python
from sgraph_ai_app_send.lambda__user.testing.Send__User_Lambda__Test_Server import (
    setup__send_user_lambda__test_server,
    setup__send_user_lambda__test_client,
    Send__User_Lambda__Test_Objs,
)
```

This is the right pattern — the QA project tests SG/Send, it doesn't fork it. But three risks exist:

**Risk 1 — Silent breakage.** The `Send__User_Lambda__Test_Server` module is effectively a public API surface. Any change to it (method signatures, return types, the `Send__User_Lambda__Test_Objs` schema) breaks QA tests with no CI signal in the main repo. The main repo's CI doesn't run QA tests.

**Risk 2 — Version drift.** The QA repo's `pyproject.toml` depends on `osbot_fast_api_serverless = "*"` — same as the main repo. Both track latest. If the main repo pins versions (as the Librarian recommends), the QA repo must follow or face mismatches.

**Risk 3 — No contract documentation.** The test server module has good inline documentation but no formal "this is a public interface" marker. A developer refactoring the main repo wouldn't know the QA repo depends on these specific exports.

**Recommendations:**

1. Add a comment block at the top of `Send__User_Lambda__Test_Server.py`: `# PUBLIC API — consumed by SG_Send__qa. Do not change signatures without updating the QA repo.`
2. Add a CI job in the main repo that pip-installs `sg-send-qa` and runs `pytest tests/unit/` from the QA repo (not the browser tests — just the unit tests that validate the API schemas and runner)
3. Version-pin the main repo's package in the QA repo's `pyproject.toml` rather than relying on `*`

## 4. The QA API Uses Vanilla FastAPI — Diverges from Main Repo Patterns

This is the most architecturally significant observation for the planned refactoring.

**Main repo pattern:** `Serverless__Fast_API` base class → `Fast_API__Routes` subclass per domain → `Type_Safe` schemas → service injection via `add_routes()` → Mangum handler for Lambda.

**QA repo pattern:** Vanilla `FastAPI()` app → `APIRouter` per layer → `dict = Body(default={})` for all request bodies → manual schema construction via `_req(body)` helper → no Serverless__Fast_API, no Fast_API__Routes, no route_path decorator.

The QA repo depends on `osbot_fast_api_serverless` (it's in `pyproject.toml`) but **only uses it for `Fast_API_Server` in the test harness** — the actual QA API server doesn't use the osbot patterns at all.

### What the QA API has today

| Component | Pattern | Notes |
|-----------|---------|-------|
| `QA_API__Server.py` | Vanilla `FastAPI()` with `app.include_router()` | No Serverless__Fast_API, hardcoded version string |
| `server/main.py` | Separate vanilla `FastAPI()` | Test runner + static site serving — completely disconnected from QA API |
| Route files (4) | `APIRouter` with `@router.post()` | All accept `dict = Body(default={})`, manually construct `Schema__QA_Request` |
| `QA_API__Runner` | `Type_Safe` | Core execution engine — well-structured |
| `QA_API__Session` | `Type_Safe` context manager | Session lifecycle — well-structured |
| Schemas (3) | `Type_Safe` | `Schema__QA_Request`, `Schema__QA_Response`, `Schema__Capture_Config` — correct pattern |
| Tests | Use `unittest.mock` | **Violates** the "no mocks" rule stated in both repos' CLAUDE.md |

### Two FastAPI apps, no connection

The QA repo has **two separate FastAPI applications** that don't know about each other:

1. `sg_send_qa/api/QA_API__Server.py` — the QA automation API (browser, upload, download, workflow routes)
2. `sg_send_qa/server/main.py` — the test runner + docs server (health, tests/run, docs, static site)

Neither extends Serverless__Fast_API. Neither uses the route class pattern. Neither is Lambda-deployable.

## 5. The QA API's Architecture Is Sound, the FastAPI Wiring Needs Alignment

The internal architecture of the QA API is well-designed:

```
Route handler (thin)
  → QA_API__Runner (trace ID, timing, error handling, capture config)
    → QA_API__Session (browser lifecycle — local or live)
      → SG_Send__Browser__Pages (page-level operations)
        → JS_Query__Execute (safe JS evaluation)
          → Playwright page
```

Each layer has a single responsibility. The session is a proper context manager. The runner handles cross-cutting concerns. The schemas use Type_Safe correctly.

The issue is only at the **FastAPI surface** — the routes accept raw dicts, manually extract fields, and construct schemas. This is the layer that should be refactored to use osbot-fast-api patterns, bringing it in line with the main repo.

## 6. Test Coverage of the QA API Itself Uses Mocks

The `test_QA_API__Runner.py` tests use `unittest.mock.patch` to avoid starting a real browser. Both the main repo and the QA repo explicitly state "no mocks, no patches" as a foundational rule.

The reason for the mocks is understandable — you don't want to start Playwright for unit tests of the runner logic. But the right pattern (following the main repo) would be to inject a mock-free alternative: either a stub `QA_API__Session` that returns canned data, or a `Type_Safe` subclass that overrides `_setup_local()` to skip browser creation.
