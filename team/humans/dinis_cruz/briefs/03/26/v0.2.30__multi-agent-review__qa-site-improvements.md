## Current State Assessment

Before reviewing the improvements, here is what exists today at `qa.send.sgraph.ai`:

- Jekyll static site with custom layout (sidebar + content)
- 8 use-case groups in the sidebar (Access Gate, Upload & Share, Separate Key, Navigation, Gallery, Browse, Viewer, PDF)

- Metadata captures method names and docstrings but not the code itself

## Architect Review

### Improvement 1: Code Alongside Screenshots — Approved with design constraints

The approach is correct — test code should be visible on the documentation page. Two implementation levels:

**Level 1 (achievable now):** Full test file source embedded on the page in a collapsible `<details>` block. `QA_Generate_Docs` already has `read_test_source()` which reads the test file — it's just not called during page generation. Wire it up.

**Level 2 (requires metadata enhancement):** Per-screenshot code snippets. This requires the `ScreenshotCapture.capture()` call to record which line of code triggered it. Python's `inspect` module can capture the caller's source location:

```python
import inspect
frame = inspect.currentframe().f_back
source_file = frame.f_code.co_filename
line_number = frame.f_lineno
```

Store `{"file": "test__combined_link.py", "line": 42}` in the metadata alongside each screenshot name. Then `QA_Generate_Docs` reads the source file, extracts the surrounding lines, and renders them inline.

**Risk:** Line numbers are fragile across refactors. Consider also storing the code snippet text at capture time (3-5 lines around the `capture()` call) rather than relying on line numbers at generation time.

**Recommendation:** Do Level 1 first (full source, collapsible). Level 2 in a follow-up.

### Improvement 2: Bidirectional GitHub Links — Approved

**QA Site → GitHub:** Straightforward. The repo URL is known (`https://github.com/the-cyber-boardroom/SG_Send__qa`), the branch is `dev`, and the test file path is derivable from the use-case name. Generate the link during `scaffold_page()`:

```
https://github.com/the-cyber-boardroom/SG_Send__qa/blob/dev/tests/qa/v030/p0__combined_link/test__combined_link.py
```

The commit hash should also be captured during CI (from `$GITHUB_SHA`) and embedded in the page footer or metadata.

**GitHub → QA Site:** Add a comment at the top of each test file with the QA site URL. This can be automated during `generate-docs` — write a companion `.qa-url` file or update the test file header. Alternatively, a badge in the repo README linking to the live site.

**Architecture note:** The `_metadata.json` should include `git_sha` and `site_url` fields. CI passes `$GITHUB_SHA` as an environment variable to `QA_Generate_Docs`.

### Improvement 3: Commit and Version Provenance — Approved with CI ordering fix

**Current CI ordering problem (confirmed):**

```
Current:
  tests pass → generate-docs → deploy-docs → increment-tag
                                                    ↑
                                            version bumps AFTER docs

Correct:
  tests pass → increment-tag → generate-docs → deploy-docs
                     ↑                              ↑
               version available          docs reference correct version
```

The `increment-tag` job currently runs in parallel with `generate-docs` — both depend on test jobs, neither depends on the other. This means generated pages may reference the old version number.

**Fix:** `generate-docs` should depend on `increment-tag`:

```yaml
generate-docs:
  needs: [run-integration-tests, run-v030-browser-tests, increment-tag]
```

But `increment-tag` has `if: inputs.should_increment_tag` — it may not run. So `generate-docs` needs `if: always()` on that dependency or a conditional check.

**Simpler alternative:** Have `generate-docs` read the version from `sg_send_qa/version` file (which `increment-tag` updates). If `increment-tag` ran first, the version is correct. If it didn't run (manual push, `should_increment_tag=false`), the existing version is fine.

### Improvement 6: Release Notes from Tests — Approved, phase it

The issue-file pattern is sound but should be introduced incrementally:

**Phase 1:** Define the issue file format (markdown, one per task/bug). Create 5 examples from the known bugs and recent work. No automation yet.

**Phase 2:** Add a `generate_release_notes` step to the doc generator that walks issue files and assembles a release page with links to tests, screenshots, and commits.

**Phase 3:** Auto-populate issue files from CI data (commit messages, PR descriptions, test results).

Phase 1 can start immediately. Phases 2-3 depend on the GitHub link infrastructure (Improvement 2).

### Improvement 7: Static Enrichment — Defer

This is Villager/Town Planner territory. The Jekyll build can do more, but the current priority is wiring up the foundations (code, links, provenance). Enrichment should wait until the base data (code on pages, GitHub links, version numbers) is in place. Pulling commit messages and PR descriptions into the build is valuable but not urgent.

### Concrete Implementation Steps

**Step 1: Add provenance to every generated page**

In `QA_Generate_Docs.scaffold_page()`, add after the front matter:

```python
import os, subprocess

git_sha   = os.environ.get('GITHUB_SHA', subprocess.getoutput('git rev-parse HEAD'))[:8]
version   = Path('sg_send_qa/version').read_text().strip()
timestamp = datetime.now(timezone.utc).strftime('%Y-%m-%d %H:%M UTC')

md += f"\n> Generated at commit [`{git_sha}`]"
md += f"(https://github.com/the-cyber-boardroom/SG_Send__qa/commit/{git_sha})"
md += f" · {version} · {timestamp}\n\n"
```

This gives every page a footer like: *Generated at commit `a1b2c3d` · v0.2.30 · 2026-03-26 00:15 UTC*

**Step 2: Add full test source (collapsible)**

`read_test_source()` already exists on `QA_Generate_Docs` but isn't called. Wire it into `scaffold_page()`:

```python
source = self.read_test_source(name)
if source:
    md += "<details>\n<summary>View test source code</summary>\n\n"
    md += f"```python\n{source}\n```\n\n"
    md += "</details>\n\n"
```

**Step 3: Add GitHub link to each page**

```python
github_base = "https://github.com/the-cyber-boardroom/SG_Send__qa/blob/dev"
test_path   = self._find_test_file(name)    # derive from use_case name
if test_path:
    md += f"[View source on GitHub]({github_base}/{test_path})\n\n"
```

The `_find_test_file()` method walks `tests/qa/v030/` looking for a directory matching the use-case name.

**Step 4: Fix CI ordering**

In `ci-pipeline.yml`, change `generate-docs` needs:

```yaml
generate-docs:
  needs: [run-integration-tests, run-v030-browser-tests, increment-tag]
  if: |
    always() &&
    needs.run-integration-tests.result != 'cancelled' &&
    needs.run-v030-browser-tests.result != 'cancelled'
```

This ensures the version file is updated before docs are generated, while still running generate-docs even if `increment-tag` was skipped.

## QA Lead Review

### Improvement 5: LLM-Assisted Review — Important, needs workflow definition

The brief says "add prompts and workflows for regular LLM review." This is valuable but under-specified. Concrete proposal:

**Screenshot analysis workflow:**
1. CI produces screenshots → published to site
2. Scheduled job (weekly or on release) collects all current screenshots
3. Sends to Claude with prompt: "Review these screenshots for visual bugs, layout issues, accessibility problems, and inconsistencies"
4. Output → new issue files in the repo

**Test coverage review workflow:**
1. Collects the list of all test files + the Schema page objects
2. Sends to Claude with prompt: "Given these page schemas and state machines, what scenarios are not covered?"
3. Output → suggested test cases as issue files

**Integration point:** This should use the QA API we're building (Task 2 from the dev brief). The API captures screenshots and page models — the LLM review consumes them. The API is the data source, the LLM review is the consumer.

**Timing:** Start after the QA API is functional (Task 2 Phases 3-4). No point reviewing screenshots manually when the API will produce structured data.

### User Story Layer (Improvement 4) — Align with QA API personas

The brief asks for a layer connecting use cases → tests → screenshots. The dev brief (v0.2.30) already defines 4 personas:

| Persona | User story | Existing tests |
|---|---|---|
| A: File Sender | Upload → share link | p0__upload__single_file, p0__combined_link, p0__friendly_token |
| B: File Recipient | Open link → view content | p0__combined_link (download half), p1__download__browse |
| C: Gallery Sender | Upload folder → gallery link | p1__upload__folder |
| D: Gallery Viewer | Open gallery → browse → save | p1__download__gallery |

The QA site should have a "User Stories" section above the use-case groups that maps each persona to their tests. This is a navigation/information-architecture change, not a code change — `QA_Generate_Docs` generates a stories index page from the persona definitions.

### Screenshot determinism (new concern)

Note: there is a separate proposal (`v0.2.30__proposal__screenshot-determinism-data-qa-mask.md`) addressing the false-positive screenshot commits. This interacts with the provenance improvement — if screenshots include commit hashes in the rendered UI, those hashes change every run, creating more noise. The `data-qa-mask` approach solves this, but it needs to be implemented alongside the provenance work.

## Designer Review

### Current Site — What Works
- Clean sidebar with grouped navigation
- Responsive layout (sidebar + content)
- Screenshot images are full-width and readable
- P0 stars provide visual priority hierarchy
- Colour scheme is neutral and professional

### Current Site — What Needs Attention

**Information density:** Each test page shows a flat list of screenshots with captions. There's no visual hierarchy between "this screenshot shows the key step" and "this is a transitional state." All screenshots are rendered at the same size and prominence.

**Recommendation:** Add screenshot categories — "key step" (full width) vs "transition" (thumbnail, expandable). The `_metadata.json` could include a `prominence` field per screenshot.

**Empty nav items:** Dashboard, Pipeline, Roadmap, and Change Requests all link to 404. These should either be populated with minimal content or hidden from the nav until ready. Broken links damage trust in the documentation.

**Code presentation:** When test source is added (Improvement 1), use a syntax-highlighted code block with line numbers. The Jekyll site should include a code highlighting library (Prism.js or Rouge — Rouge is built into Jekyll).

**Recommendation:** Rouge is already bundled with Jekyll. Add to `_config.yml`:
```yaml
highlighter: rouge
markdown: kramdown
kramdown:
  syntax_highlighter: rouge
```

**Per-screenshot code snippets** (Improvement 1, Level 2) should be rendered in a side-by-side or stacked layout:

```
┌────────────────────────────────────────┐
│ Code: sg_send.upload__set_file(...)    │  ← code snippet, syntax highlighted
├────────────────────────────────────────┤
│ [Screenshot: 02_file_selected.png]     │  ← full-width screenshot
│                                        │
│                                        │
└────────────────────────────────────────┘
```

Not inline — the code block should be above the screenshot, visually connected. This reads as "action → result."
