## Task 1: Wire Schema Page Objects into SG_Send__Browser__Pages

### Current State

Five `Schema__*Page` classes exist in `sg_send_qa/browser/`:

```
Schema__Upload_Page.py     — state, file_name, share_link, friendly_token, is_gate_visible
Schema__Download_Page.py   — state, is_key_input_visible, content_text, error_message, transfer_id
Schema__Browse_Page.py     — state, content_text, error_message, transfer_id, friendly_token
Schema__Gallery_Page.py    — state, file_count, content_text, error_message, transfer_id
Schema__Viewer_Page.py     — state, file_name, content_text, error_message, transfer_id
```

**Problem:** they have unit tests for defaults and serialisation but are NOT imported or used anywhere in `SG_Send__Browser__Pages`, the test harness, or any test file. They are dead code.

### What To Do

Add an `extract_page_model()` method to `SG_Send__Browser__Pages` for each page. These methods use `self.js()` (the JS_Query layer) to read the current page state and return a populated schema:

```python
def extract__upload_page(self) -> Schema__Upload_Page:
    return Schema__Upload_Page(
        state           = self.upload_state()                                    or '',
        file_name       = self.js().text("upload-step-select", "#file-name")     or '',
        share_link      = self.js().text("upload-step-done", "#combined-link")   or '',
        friendly_token  = self.js().text("upload-step-done", "#simple-token")    or '',
        is_gate_visible = self.is_access_gate_visible()                              ,
    )

def extract__download_page(self) -> Schema__Download_Page:
    return Schema__Download_Page(
        state               = self.download_state()                              or '',
        is_key_input_visible= self.js().light_visible("#key-input")                  ,
        content_text        = self.visible_text()                                or '',
        error_message       = self.js().light_text(".error-message")             or '',
        transfer_id         = ''                                                     ,  # extract from URL hash
    )

def extract__browse_page(self) -> Schema__Browse_Page:
    return Schema__Browse_Page(
        state           = self.download_state()                                  or '',
        content_text    = self.visible_text()                                    or '',
        error_message   = self.js().light_text(".error-message")                 or '',
        transfer_id     = ''                                                         ,  # extract from URL hash
        friendly_token  = ''                                                         ,
    )
```

The exact selectors for `file_name`, `error_message`, etc. need to be confirmed against the live DOM — use `headless=False` debug mode to inspect. The pattern is: one `extract__*` method per page, returns a Type_Safe schema, uses only `self.js()` for reads.

**Tests should then assert against schemas:**

```python
# BEFORE (current pattern):
assert self.sg_send.upload_state() == 'complete'
assert self.sg_send.is_access_gate_visible() is False

# AFTER (schema pattern):
model = self.sg_send.extract__upload_page()
assert model.state           == 'complete'
assert model.is_gate_visible is False
assert model.share_link      != ''
```

### Acceptance

- [ ] Each of the 5 `extract__*` methods exists on `SG_Send__Browser__Pages`
- [ ] Each method returns the corresponding `Schema__*Page` instance populated from the live DOM
- [ ] At least one integration test per page uses the schema assertion pattern
- [ ] Schemas are JSON-serialisable (for the API response and state capture files)

## Task 2: Create the QA API (FastAPI)

### Design Philosophy: Maximum Granularity (Explorer Phase)

We are in the Explorer phase of this API. **Over-index on exposing more endpoints.** Every action, every page, every workflow gets its own route. Later (Villager/Town Planner) we consolidate and prune. For now, more routes = more composability = more power for debugging and testing.

**Stateless by design:** Every request is completely self-contained. No state is reused between requests. No session cookies, no server-side browser pools, no "continue from where I left off." Each request starts a fresh browser, executes, captures results, tears down. This is because the API will eventually run on Lambda functions where we cannot control which instance handles which request.

**Secrets provided by the caller:** The API never stores or caches secrets. Any credentials needed for execution — access tokens, simple tokens, API keys for target servers — must be provided in each request body. The API uses them for that single request and discards them. In local mode the harness auto-generates test tokens; in live mode the caller provides them. This keeps the API stateless and avoids secrets leaking between requests or persisting on disk.

**All methods are POST:** Every endpoint receives a request body containing the execution config and capture config. Even "read" operations (get upload state, get visible text) are POST because they carry the target/auth/capture configuration. There are no GET endpoints except `/api/qa/health`.

**Capture config on every request:** The caller specifies what data to collect. The service captures everything requested and returns it in the response. This is the key to making the API useful for different consumers (QA wants screenshots, SRE wants Prometheus metrics, DevOps wants traces).

The API exposes four layers of granularity:

```
Layer 0: Raw browser actions     /api/browser/*       (open page, click, type, read)
Layer 1: Page primitives         /api/page/*          (navigate to upload page, set file, read state)
Layer 2: Feature operations      /api/upload/*        (upload with combined link, upload with token)
                                 /api/download/*      (download via browse, download via gallery)
Layer 3: User story workflows    /api/workflow/*       (Persona A: full send, Persona B: full receive)
```

Each higher layer composes the lower layers. Tests can call any layer directly.

**Auth:** All endpoints are protected by an API key header. This is provided by default by `Fast_API__Serverless` which we are using for the Lambda deployment — no custom auth middleware needed. No confidential data on any test/target server; this is purely to prevent accidental access.

**Tracing:** Every request accepts an optional `X-QA-Trace-ID` header. This trace ID is propagated to the browser session via a custom request header, so server-side logs can be correlated with QA API calls. If not provided, the API generates one.

### Two Execution Contexts

**Local mode** (`target: "local"`) — uses `SG_Send__Browser__Test_Harness`:
```
POST /api/upload/combined  { "target": "local", "filename": "test.txt" }
→ harness.setup() → upload workflow → harness.teardown() → JSON response
```

**Live mode** (`target: "https://send.sgraph.ai"`) — browser points at live URL:
```
POST /api/upload/combined  { "target": "https://send.sgraph.ai", "access_token": "...", "filename": "test.txt" }
→ browser.open(target) → inject token → upload workflow → JSON response
```

The ONLY difference: local starts servers and auto-generates the token; live takes a URL and token from the request.

Every Layer 0 request follows the same stateless pattern:

1. **`target`** — where to run (local or live URL + access token)
2. **`capture`** — what data to collect (`Schema__Capture_Config`)
3. **`page`** — which page to navigate to before performing the action
4. **`state`** *(optional)* — localStorage / cookies to inject after navigation, before action
5. **Action-specific fields** — selector, value, host/inner, actions array, etc.

The response always includes a `result` object with the action-specific output, plus whatever additional data was requested in the `capture` config.

The `/api/browser/execute` endpoint is the chained commands pattern — receives an array of actions, executes them sequentially on the same page within a single browser session, returns results for each. This enables complex sequences without multiple HTTP round-trips while remaining stateless (the browser is created and destroyed within this single request).

### Common Request/Response Structure

Every endpoint is POST and receives a body with three sections: **execution config** (what to do), **target config** (where to do it), and **capture config** (what to return).

#### Request Body (all POST endpoints)

```json
{
  "target": {
    "url"          : "local" | "https://send.sgraph.ai",
    "access_token" : "..."
  },

  "execution": {
    "mode"         : "smoke" | "qa",
    "trace_id"     : "req-12345-abc",
    "options"      : { }
  },

  "capture": {
    "execution_result"  : true,
    "page_model"        : true,
    "screenshots"       : false,
    "screenshot_on_fail": true,
    "video"             : false,
    "pdf"               : false,
    "browser_console"   : false,
    "browser_trace"     : false,
    "python_console"    : false,
    "python_trace"      : false,
    "performance_data"  : true,
    "prometheus"        : true,
    "network_log"       : false
```

The `capture` object tells the service what data to collect. Every field defaults to a sensible value per mode (`smoke` mode defaults to minimal capture; `qa` mode defaults to screenshots + traces). The caller can override any field.

**Capture config details:**

| Field | What it captures | Format |
|---|---|---|
| `execution_result` | Pass/fail status, step timing, error messages | JSON |
| `page_model` | `Schema__*Page` snapshot of final page state | JSON (Type_Safe serialised) |
| `screenshots` | Screenshot at each step (QA mode) or final state only (smoke) | base64 PNG array |
| `screenshot_on_fail` | Screenshot only when a step fails (useful for smoke mode) | base64 PNG |
| `video` | Full browser session recording (Playwright `recordVideo`) | base64 webm |
| `pdf` | PDF export of the final page state | base64 PDF |
| `browser_console` | Browser console.log/warn/error messages captured during execution | JSON array of `{level, text, timestamp}` |
| `browser_trace` | Playwright trace file (network, DOM snapshots, screenshots) | base64 zip (viewable at trace.playwright.dev) |
| `python_console` | Python stdout/stderr captured during execution (using osbot-utils `Stdout`/`Stderr`) | string |
| `python_trace` | Python traceback if execution fails | string |
| `performance_data` | Per-step duration, total duration, browser performance metrics | JSON |
| `prometheus` | Prometheus exposition format metrics for Grafana | string (text/plain) |
| `network_log` | All HTTP requests/responses made by the browser during execution | JSON array of `{url, method, status, duration_ms}` |

#### Stateless Lifecycle (per request)

Every API request follows this lifecycle — nothing survives between requests:

```
Request arrives
  │
  ├─ Parse request body (target, execution, capture)
  ├─ Generate trace_id if not provided
  │
  ├─ Start capture collectors (based on capture config):
  │   ├─ Stdout/Stderr capture (python_console)
  │   ├─ Browser console listener (browser_console)
  │   ├─ Network request logger (network_log)
  │   └─ Video recording (video — configured at browser launch)
  │
  ├─ Create browser session:
  │   ├─ LOCAL:  SG_Send__Browser__Test_Harness().headless(True).setup()
  │   └─ LIVE:   SG_Send__Browser__Pages(target=url, access_token=token)
  │
  ├─ Set tracing header: X-QA-Trace-ID on all browser requests
  │
  ├─ Execute workflow/action (timed per step)
  │
  ├─ Collect outputs (based on capture config):
  │   ├─ page_model → extract__*_page()
  │   ├─ screenshots → already captured during execution
  │   ├─ browser_trace → page.context.tracing.stop(path=...)
  │   ├─ prometheus → format metrics
  │   └─ pdf → page.pdf()
  │
  ├─ Tear down browser session (kill Chrome, stop servers if local)
  │
  └─ Return JSON response with requested data
```

**Nothing persists.** No browser pool. No session cache. No state between requests. Every request pays the full setup cost (~0.35s local, ~0s live) and gets a clean environment. This is the Lambda-compatible design.

### Tracing Header Propagation

When a `trace_id` is provided (or auto-generated), the API:

1. Sets it as a custom header on every browser request: `X-QA-Trace-ID: req-12345-abc`
2. Returns it in the response so the caller can correlate
3. Server-side logs on the target (SG/Send) can filter by this header to see exactly which API calls were triggered by which QA run

```python
# In the browser session setup:
sg_send.raw_page().set_extra_http_headers({
    'X-QA-Trace-ID': trace_id
})
```

This enables: "QA API run `req-12345-abc` failed at step 3 — here are the server logs for that exact request."

## Task 3: Migrate v030 Tests to Harness Infrastructure

### Current State

The v030 tests use a completely different infrastructure from the integration tests:

| Aspect | Integration tests | v030 tests |
|---|---|---|
| Base class | `unittest.TestCase` | pytest classes (no base) |
| Server lifecycle | `SG_Send__Browser__Test_Harness` | `conftest.py` fixtures (`subprocess.Popen` + polling loop) |
| Browser | `QA_Browser` (CDP, persistent) | `sync_playwright()` (fresh per session) |
| Page interaction | `SG_Send__Browser__Pages` methods | Raw `page.locator()` |
| JS evaluation | `JS_Query__Shadow_DOM` | Raw `page.evaluate()` / `page.text_content()` |
| Waits | Event-based (predicates) | Mix of event-based (`browser_helpers.py`) and `wait_for_load_state("networkidle")` |
| State queries | `self.sg_send.upload_state()` | `page.text_content("body")` string search |
| Screenshots | Not yet | `ScreenshotCapture` fixture |

**The goal is NOT to rewrite all v030 tests at once.** It's an incremental migration where each test file moves to the new infrastructure one at a time.

### Migration Strategy

#### Phase 0: Convert v030 test classes to unittest.TestCase (do first!)

Before changing any infrastructure, convert v030 test classes from raw pytest classes to `unittest.TestCase`. This is the same pattern as the integration tests and gives us `setUpClass`/`tearDownClass` for shared state.

**Why first:** the rest of the migration (harness, sg_send, shared browser) depends on `setUpClass` for lifecycle management. Pytest's function-scoped `page` fixture creates a fresh page per test — `TestCase.setUpClass` lets us share a browser session across an entire class, matching the integration test pattern.

**Current v030 pattern (pytest class, no base):**
```python
class TestSingleFileUpload:
    def test_upload_page_loads(self, page, ui_url, screenshots):
        goto(page, f"{ui_url}/en-gb/")
        ...

    def test_single_file_upload_flow(self, page, ui_url, send_server, screenshots):
        ...
```

**Target pattern (TestCase with setUpClass):**
```python
class test_Single_File_Upload(TestCase):
    @classmethod
    def setUpClass(cls):
        cls.harness = SG_Send__Browser__Test_Harness().headless(True).setup()
        cls.sg_send = cls.harness.sg_send
        with cls.harness as _:
            cls.access_token = _.set_access_token()

    @classmethod
    def tearDownClass(cls):
        cls.harness.teardown()

    def test__01__upload_page_loads(self):
        with self.sg_send as _:
            _.page__root()
            assert _.is_upload_zone_visible() or _.is_access_gate_visible()

    def test__02__single_file_upload_flow(self):
        with self.sg_send as _:
            _.page__root()
            _.upload__set_file("test.txt", b"Hello from QA")
            ...
```

Key changes in this phase:
- Class inherits `TestCase`
- Class name follows `test_*` convention (lowercase with underscores, matching integration tests)
- `setUpClass` / `tearDownClass` replace fixtures
- Tests are numbered (`test__01__`, `test__02__`) for ordered execution within a class
- No fixture parameters (`self.sg_send` replaces `page`, `ui_url`, `send_server`)
- Screenshots: `self.screenshots.capture(self.sg_send.raw_page(), ...)` (ScreenshotCapture becomes a class attribute set up in `setUpClass`)

**This phase does NOT change the conftest fixtures yet** — the conftest can still provide `send_server`, `ui_url` etc. for non-migrated tests. The `TestCase` classes just don't use them.

#### Phase 1: Add `sg_send` fixture to conftest (bridge period)

For any tests that haven't been converted to TestCase yet, add `sg_send` as a pytest fixture alongside the existing ones:

```python
@pytest.fixture(scope="session")
def sg_send(ui_server, send_server):
    """SG_Send__Browser__Pages wired to the existing v030 UI server."""
    from sg_send_qa.browser.SG_Send__Browser__Pages import SG_Send__Browser__Pages
    port = int(ui_server.split(":")[-1].rstrip("/"))
    instance = SG_Send__Browser__Pages(headless=True, target_port=port)
    instance.page__qa_setup()
    instance.storage__set_token(send_server.access_token)
    yield instance
    instance.qa_browser().stop()
```

This is the bridge: non-migrated tests still use `page` fixture, migrated tests use `TestCase` with harness (preferred) or `sg_send` fixture.

#### Phase 2: Migrate P0 tests (highest value)

Start with the 3 P0 browser test files that were just refactored to use `browser_helpers.py`:

| File | Current approach | Migration |
|---|---|---|
| `p0__upload__single_file` | `page.locator("#file-input").set_input_files()` | `sg_send.upload__set_file()` |
| `p0__combined_link` | Raw locators + `browser_helpers.wait_for_upload_state()` | `sg_send.upload__*` + `sg_send.upload__get_combined_link()` |
| `p0__friendly_token` | Raw locators + `browser_helpers.wait_for_upload_state()` | `sg_send.upload__*` + `sg_send.upload__get_friendly_token()` |

The `browser_helpers.py` functions (`goto`, `handle_access_gate`, `wait_for_upload_state`) become unnecessary once `SG_Send__Browser__Pages` is wired — those methods already exist on `sg_send`.

**Example migration for `p0__upload__single_file`:**

```python
# BEFORE (current — raw pytest class + fixtures):

        page.locator('[data-mode="combined"]').click()
        wait_for_upload_state(page, "confirming")
        page.locator("#upload-next-btn").click()
        page.locator("a[href*='#']").first.wait_for(...)

# AFTER (TestCase + harness + sg_send):
class test_Single_File_Upload(TestCase):
    @classmethod
    def setUpClass(cls):
        cls.harness     = SG_Send__Browser__Test_Harness().headless(True).setup()
        cls.sg_send     = cls.harness.sg_send
        cls.screenshots = ScreenshotCapture(...)
        with cls.harness as _:

    @classmethod

        cls.harness.teardown()

    def test__01__single_file_upload_flow(self):
        with self.sg_send as _:

            _.upload__set_file("test.txt", b"Hello from QA")
```

Notice:
- `TestCase` with `setUpClass` — one harness per class, shared browser

1. **No hardcoded `wait(N)` / `sleep()` / `wait_for_timeout()`** — see `v0.2.20__tech-notes__wait-elimination-and-gh-warnings.md` Section 0
