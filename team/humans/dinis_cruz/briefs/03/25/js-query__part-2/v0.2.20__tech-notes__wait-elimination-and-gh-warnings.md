## 0. MANDATORY POLICY: Hardcoded Waits Are Banned

**All hardcoded `wait(ms)` / `sleep()` / `wait_for_timeout()` calls are banned from the codebase.** This is not a guideline — it is a hard rule. There are no exceptions without explicit approval and documentation.

If you find yourself writing `self.sg_send.wait(N)`, **stop**. Something is wrong with the infrastructure, not with the test. Fix the infrastructure.

### The rule

1. **Never** add `wait(N)`, `sleep(N)`, `page.wait_for_timeout(N)`, or any hardcoded millisecond delay to any test or framework code.
2. **Always** use event-based waits: `wait_for_page_ready()`, `wait_for_upload_state()`, `wait_for_function()`, `locator.wait_for()`, or Playwright's built-in auto-waiting.
3. **If** no event-based alternative exists, this is a bug in the application or the framework — fix it there first (Principle 3: fix problems in the application, not the tests).
4. **If** after exhausting all alternatives a hardcoded wait is genuinely the only option, it **must** have:
   - A comment explaining **exactly** what DOM event or state change it is waiting for
   - A comment explaining **why** no event-based wait is possible
   - A TODO linking to the fix that will eliminate it
   - Approval from the project lead

### Why this matters

Hardcoded waits are:
- **Slow** — they always wait the full duration even when the condition is met in 10ms. 19 waits × 2 seconds average = 38 seconds of dead time per test run.
- **Flaky** — they pass on fast machines and fail on slow CI runners. They are the #1 cause of "works on my machine" in browser automation.
- **Contagious** — once one `wait(1500)` appears, every subsequent developer copies the pattern. The 19 waits in this commit were all added in one session because the first one "worked."
- **Hiding bugs** — a `wait(4000)` after decrypt masks the real question: "what signal does the page emit when decrypt is complete?" If the answer is "nothing," that's a bug in the application. Fix the application. Add a `data-decrypt-complete` attribute. Then wait for it.

### The alternatives (in order of preference)

1. **`wait_for_page_ready()`** — waits for `body[data-ready]`. Use after every navigation.
2. **`wait_for_upload_state(state)`** — waits for the upload wizard state machine. Use during upload flows.
3. **`raw_page().wait_for_function(js_predicate)`** — waits for any JS condition to become truthy. Use with `JS_Query__Shadow_DOM.predicate__*` methods for Shadow DOM conditions.
4. **`locator.wait_for(state="visible")`** — Playwright's built-in wait for element visibility. Use for specific elements.
5. **`wait_for_component(tag_name)`** — waits for a Web Component to be attached to the DOM. Use for lazy-loaded components.
6. **Fix the application** — add a `data-ready`, `data-state`, or custom event to the component. Then wait for it. This is Principle 3.

## 1. The Wait Problem — 19 Hardcoded Waits

The new tests in `test_SG_Send__Browser__Pages__Download.py` (9 waits), `test_SG_Send__Browser__Pages__Access_Gate.py` (7 waits), and `test_SG_Send__Browser__Pages__Workflow.py` (3 waits) all use `self.sg_send.wait(N)` — hardcoded millisecond delays.

This violates Principle 4 from the project's own QA API Principles brief (v0.16.53):

> "Hardcoded waits are always a code smell. They signal that something about the test infrastructure or the application is not right."

### Why the waits exist

The `open()` method was changed to `wait_until="commit"` (fires on first byte received). This means `page__root()` returns **before JS has even started executing**. Every caller then needs a `wait()` to compensate.

### The root cause

The page already signals when it's ready — `body[data-ready="true"]` is set by the app's init code. We have `wait_for_page_ready()` which waits for exactly this. But nobody calls it because the waits are "easier."

### The fix: event-based waits using signals already on the page

Every `wait(N)` maps to a specific DOM event. Here are the replacements:

#### Access Gate tests (7 waits → 0)

```python
# BEFORE (5 instances of this pattern):
self.sg_send.page__root()
self.sg_send.wait(1500)                                    # guessing page rendered
assert self.sg_send.is_access_gate_visible() is True

# AFTER:
self.sg_send.page__root()
self.sg_send.wait_for_page_ready()                         # event: body[data-ready]
assert self.sg_send.is_access_gate_visible() is True
```

```python
# BEFORE (2 instances after submit):
self.sg_send.access_gate__enter_and_submit(token)
self.sg_send.wait(1500)                                    # guessing gate responded

# AFTER:
self.sg_send.access_gate__enter_and_submit(token)
# gate either dismisses (upload zone appears) or shows error (gate stays)
# both are instant DOM changes — no need to wait
# is_access_gate_visible() and is_upload_zone_visible() already have timeout=2000
```

#### Download tests (9 waits → 0)

```python
# BEFORE:
self.sg_send.page__download()
self.sg_send.wait(2000)                                    # guessing send-download rendered
is_visible = self.sg_send.raw_page().locator("#entry-input").is_visible(timeout=5000)

# AFTER:
self.sg_send.page__download()
self.sg_send.wait_for_page_ready()                         # event: body[data-ready]
is_visible = self.sg_send.raw_page().locator("#entry-input").is_visible(timeout=5000)
# locator.is_visible already has its own timeout — no pre-wait needed
```

```python
# BEFORE (browse/gallery with hash — 7 instances):
self.sg_send.page__browse_with_hash(tid, key_b64)
self.sg_send.wait(4000)                                    # guessing decrypt finished
text = self.sg_send.visible_text()

# AFTER — Option A (simple, uses data-ready):
self.sg_send.page__browse_with_hash(tid, key_b64)
self.sg_send.wait_for_page_ready()                         # page init complete
# Then wait for the specific content to appear:
self.sg_send.raw_page().wait_for_function(
    "document.body.innerText.includes('expected content')",
    timeout=10000
)

# AFTER — Option B (uses JS_Query__Shadow_DOM predicates):
self.sg_send.page__browse_with_hash(tid, key_b64)
self.sg_send.wait_for_page_ready()
query = JS_Query__Shadow_DOM(raw_page=self.sg_send.raw_page())
self.sg_send.raw_page().wait_for_function(
    query.predicate__exists("send-download", "#decrypted-content"),
    timeout=10000
)
```

```python
# BEFORE (separate key decrypt):
self.sg_send.download__enter_key(key_b64)
self.sg_send.wait(500)
self.sg_send.download__click_decrypt()
self.sg_send.wait(3000)

# AFTER:
self.sg_send.download__enter_key(key_b64)
self.sg_send.download__click_decrypt()
# The decrypt button click triggers async pipeline — wait for result:
# Either content appears, or page state changes, or URL updates
# Use wait_for_function on whatever signal the app produces
```

#### Workflow tests (3 waits → 0)

Same patterns as above — `wait_for_page_ready()` after navigation, `wait_for_function()` for async results.

### The `open()` method — `wait_until="commit"` is too weak

The change to `wait_until="commit"` was made because SG/Send's inline scripts "may block domcontentloaded." But `commit` fires before any JS executes, which forces every caller to add their own wait.

Recommended: revert to `wait_until="domcontentloaded"` (the Playwright default), then use `wait_for_page_ready()` when needed. Or keep `commit` but make `wait_for_page_ready()` the standard follow-up in the `SG_Send__Browser__Pages.open()` method itself:

```python
def open(self, path, use_language=True, wait_for_ready=True):
    url = self.url__for_path(path=path, use_language=use_language)
    self.raw_page().goto(url, wait_until="commit")
    if wait_for_ready:
        self.wait_for_page_ready()
    return self.page()
```

This way callers never need to manually wait after navigation. The `wait_for_ready=False` escape hatch is there for special cases (like qa-setup.html which doesn't need JS init).

### The principle: reads don't wait, navigation waits for ready

The design from the `JS_Query__Shadow_DOM` brief (in `team/humans/dinis_cruz/briefs/03/25/js-query__part-1/`) and the updated two-class architecture (in `team/humans/dinis_cruz/briefs/03/25/js-query__part-2/`) separates concerns:

1. **Navigation methods** → wait for page ready (event-based, `body[data-ready]`)
2. **Pure reads** (`shadow_text`, `exists`, `visible`) → instant, return None if missing
3. **Predicates** (`predicate__exists`, `predicate__text`) → return JS strings for `wait_for_function`
4. **Tests** → compose: navigate → (optionally wait for specific condition) → read → assert

No method bakes in a hardcoded delay. The test owns the timing decision.

## 3. Additional Notes from Code Review

### XSS in `js_evaluate` calls

Multiple tests use raw string interpolation into JS:

```python
self.sg_send.js_evaluate(f"localStorage.setItem('qa-transfer-id', '{tid}')")
```

The `tid` and `key_b64` values come from the test server (safe), but this pattern shouldn't be copied to production code. The `JS_Query__Shadow_DOM` class uses `str_to_base64` + `atob()` — all localStorage operations should go through that or through the `storage__set_token` pattern.

### `transfer_helper()` — good addition

The `harness.transfer_helper()` method enables API-level test data setup (Principle 5: "Set up state directly, do not replay entire workflows"). The download tests use it to create transfers without running the upload wizard. This is the right pattern.

### `_open_with_hash()` — good fix

Hash fragments were being sanitised by `url_join_safe()`. The explicit URL construction for hash-based routes is correct. The `wait_until="commit"` concern applies here too — these methods should either use `domcontentloaded` or be followed by `wait_for_page_ready()`.

## 4. MANDATORY: All JS Communication Goes Through `JS_Query__Shadow_DOM`

The classes `JS_Query__Execute` and `JS_Query__Shadow_DOM` were designed specifically for this project. They exist in two iterations:

- **Part 1** (original): `team/humans/dinis_cruz/briefs/03/25/js-query__part-1/JS_Query__Shadow_DOM.py` — single class with everything
- **Part 2** (refactored): `team/humans/dinis_cruz/briefs/03/25/js-query__part-2/` — split into two classes:
  - `JS_Query__Execute.py` — base layer: safe eval, base64 encoding, localStorage, light DOM reads
  - `JS_Query__Shadow_DOM.py` — inherits Execute, adds shadow traversal + predicates

**Part 2 is the canonical version.** The two-class architecture separates the execution layer (how we talk to JS) from the query layer (how we traverse Shadow DOM):

```
JS_Query__Execute                    ← owns the pipe to JS
├── evaluate()                       ← the ONE place raw_page.evaluate is called
├── evaluate_iife()                  ← wraps in (() => { ... })()
├── eval_safe()                      ← {placeholder} → atob('base64')
├── storage_get/set/remove/clear/dump
├── light_text/value/attribute/property/exists/visible
│
└── JS_Query__Shadow_DOM             ← inherits everything above, adds shadow
    ├── _js_resolve(*selectors)      ← builds N-level shadow traversal JS
    ├── text/html/attribute/value/exists/visible/count/texts
    └── predicate__exists/text/text_equals/visible
```

**All Python→JS communication must go through these classes.** No exceptions.

### What the current code does (wrong)

The CCW session code uses raw JS everywhere:

```python
# Raw js_evaluate with string interpolation — XSS vulnerable, no Shadow DOM awareness
self.sg_send.js_evaluate(f"localStorage.setItem('qa-transfer-id', '{tid}')")
self.sg_send.js_evaluate("localStorage.getItem('qa-transfer-id')")

# Raw locator for Shadow DOM text — unreliable (known failure mode)
el = self.raw_page().locator("upload-step-done #simple-token")
el.text_content()

# Inline JS for Shadow DOM traversal — duplicated, not base64 protected
self.sg_send.js_evaluate("""
    document.querySelector("upload-step-done").shadowRoot.querySelector("#simple-token").innerText
""")

# Raw locator for visibility checks
self.sg_send.raw_page().locator("#entry-input").is_visible(timeout=5000)
```

### What the code should do (correct)

```python
# All reads go through JS_Query__Shadow_DOM
query = JS_Query__Shadow_DOM(raw_page=self.sg_send.raw_page())

# Shadow DOM text extraction — base64 protected, arbitrary depth, returns None if missing
text = query.text("upload-step-done", "#simple-token")

# Light DOM reads
title = query.light_text("title")
ready = query.light_attribute("body", "data-ready")

# Existence and visibility checks
exists  = query.exists("upload-step-select", "#file-input")
visible = query.visible("upload-step-select", "#drop-zone")

# Multiple matches
tokens = query.texts("upload-step-done", ".share-box")
count  = query.count("upload-step-done", ".share-box")

# localStorage — through eval_safe or dedicated helper (base64 protected)
# NOT through raw js_evaluate with string interpolation

# Predicates for event-based waits (Section 0)
pred = query.predicate__exists("upload-step-done", "#simple-token")
self.sg_send.raw_page().wait_for_function(pred, timeout=10000)
text = query.text("upload-step-done", "#simple-token")
```

### Why this matters

1. **XSS protection** — `JS_Query__Shadow_DOM` base64-encodes all values via `str_to_base64()` / `atob()`. Raw `js_evaluate(f"...'{variable}'...")` is injection-vulnerable. Every raw interpolation is a security hole.

2. **Shadow DOM reliability** — Playwright's `locator.text_content()` doesn't reliably extract text from shadow roots (we already hit this with `#simple-token`). `JS_Query__Shadow_DOM.text()` uses explicit `shadowRoot.querySelector()` which always works.

3. **Multi-level traversal** — the variadic `*selectors` pattern handles any depth:
   ```python
   query.text("send-upload", "upload-step-done", "#simple-token")  # 2 shadow boundaries
   ```
   No need to write custom JS for each nesting level.

4. **Predicates** — the `predicate__*` methods return JS function strings for `wait_for_function()`. This is how we eliminate hardcoded waits (Section 0). Without `JS_Query__Shadow_DOM`, every wait replacement requires hand-written JS.

5. **Single responsibility** — one class owns all JS evaluation. When the JS interface changes (new shadow components, new attributes, new state signals), there's one place to update.

### The rule

- **No** `raw_page().evaluate()` in test code
- **No** `js_evaluate()` with string interpolation in test code
- **No** `raw_page().locator()` for reading text/attributes from Shadow DOM
- **All** JS reads, writes, checks, and predicates go through `JS_Query__Shadow_DOM`
- **Extend** `JS_Query__Shadow_DOM` when new capabilities are needed — don't bypass it

Both classes should be moved from `team/humans/dinis_cruz/briefs/03/25/js-query__part-2/` to `sg_send_qa/browser/` and integrated into `SG_Send__Browser__Pages` as the standard JS evaluation layer. `SG_Send__Browser__Pages` should hold a `JS_Query__Shadow_DOM` instance (which inherits all `JS_Query__Execute` capabilities) and delegate all JS operations to it.

## Summary

| Issue | Severity | Fix |
|---|---|---|
| **19 hardcoded `wait()` calls** | **Critical — policy violation** | **Remove all. Replace with event-based waits. See Section 0.** |
| **Raw JS evaluation everywhere** | **Critical — policy violation** | **All JS must go through `JS_Query__Execute` / `JS_Query__Shadow_DOM` (part-2). See Section 4.** |
| `open()` uses `wait_until="commit"` | High | Either revert to `domcontentloaded` or add `wait_for_page_ready()` inside `open()` |
| `target_url` in dev/main callers | Low | Remove from both YAML files |

The wait ban and the `JS_Query__Execute` / `JS_Query__Shadow_DOM` mandate are both non-negotiable. They solve the same root problem from opposite ends: waits are guessing at what the DOM is doing; raw JS is bypassing the safe, tested, Shadow DOM-aware query layer that already exists. Use the tools we built. They are in `team/humans/dinis_cruz/briefs/03/25/js-query__part-2/`.
