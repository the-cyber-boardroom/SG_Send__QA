**Version:** v0.2.20
**Date:** 25 March 2026
**From:** Librarian (QA Explorer Team)
**To:** Explorer Team (all roles)
**Type:** Correction brief
**Triggered by:** `@dataclass` usage in architecture plan (7cba806) and dev brief (`v0.16.61__dev-brief__playwright-optimization.md`)

## The Problem

Two documents propose `@dataclass` for the new page schema classes:

**1. Architecture plan** (`docs/architecture/v0.2.20__test-architecture-and-implementation-plan.md`):
```python
# ❌ WRONG — line 120
@dataclass
class Schema__Upload_Page:
    step: str
    filename: str | None
    ...
```

**2. Dev brief** (`team/humans/dinis_cruz/briefs/03/25/v0.16.61__dev-brief__playwright-optimization.md`):
```python
# ❌ WRONG — lines 74, 82
@dataclass
class UploadPageModel:
    step: str
    mode: str
    ...
```

**This is incorrect.** The project standard is `Type_Safe` from `osbot-utils`. This was established as non-negotiable in the refactoring brief pack (Doc 02, §3): *"All Python code in the QA repo must use Type_Safe from osbot-utils. No Pydantic. No raw dicts for structured data."*

The entire Phase 1-5 refactoring was built on this requirement. Every class in `sg_send_qa/` already uses Type_Safe. Introducing `@dataclass` would create two competing patterns.

---

## What to Use Instead

### The Import

```python
# Current codebase uses two import paths (both work):

# Newer (preferred — used in browser/ package):
from osbot_utils.type_safe.Type_Safe import Type_Safe

# Older alias (used in models/ and utils/ — works, will be updated):
from osbot_utils.base_classes.Kwargs_To_Self import Kwargs_To_Self as Type_Safe
```

### Basic Pattern — Fields with Defaults

```python
# ❌ @dataclass
@dataclass
class UploadPageModel:
    step: str = "idle"
    filename: str | None = None
    filesize_bytes: int = 0

# ✅ Type_Safe
class Schema__Upload_Page(Type_Safe):
    step           : str  = "idle"
    filename       : str  = ""        # use "" not None (Type_Safe convention)
    filesize_bytes : int  = 0
```

**Key differences from @dataclass:**
- No `@dataclass` decorator — just inherit from `Type_Safe`
- No `__init__` needed — Type_Safe auto-generates it from annotations
- No `| None` — use empty string `""` or `0` as defaults instead of `None`
- Constructor accepts kwargs: `Schema__Upload_Page(step="complete", filename="test.txt")`

### Collections — list, dict

```python
class Schema__Upload_Page(Type_Safe):
    step        : str  = "idle"
    test_files  : list              # empty list by default (Type_Safe handles this)
    mode_cards  : dict              # empty dict by default
```

**Type_Safe handles mutable defaults safely.** Unlike `@dataclass` (which requires `field(default_factory=list)`), Type_Safe gives each instance its own copy automatically. No shared-state bugs.

### Nested Type_Safe Objects

```python
class Schema__Gallery_Card(Type_Safe):
    filename    : str  = ""
    thumbnail   : str  = ""
    file_type   : str  = ""

class Schema__Gallery_Page(Type_Safe):
    cards       : list              # list of Schema__Gallery_Card instances
    total_files : int  = 0
    view_mode   : str  = "gallery"
    share_link  : str  = ""
```

### Methods on Type_Safe Classes

Type_Safe classes support regular methods. This is how the existing codebase works — `QA_Transfer_Helper`, `SG_Send__Browser__Test_Harness`, `QA_Generate_Docs` all have methods:

```python
class Schema__Upload_Page(Type_Safe):
    step           : str  = "idle"
    filename       : str  = ""
    filesize_bytes : int  = 0
    done_link      : str  = ""

    def is_complete(self) -> bool:
        return self.step == "complete" and self.done_link != ""

    def has_file(self) -> bool:
        return self.filename != ""
```

### JSON Serialisation

Type_Safe provides `json()` for serialisation:

```python
model = Schema__Upload_Page(step="complete", filename="test.txt")

# Serialise to dict
data = model.json()
# → {"step": "complete", "filename": "test.txt", "filesize_bytes": 0, "done_link": ""}

# Write to disk
import json
Path("state.json").write_text(json.dumps(data, indent=2))

# Reconstruct from dict
model2 = Schema__Upload_Page(**data)
assert model2.step == "complete"
```

### Composition — Type_Safe as Fields

```python
class Schema__Browser_Test_Config(Type_Safe):
    headless       : bool                = True
    capture_stderr : bool                = True
    host           : Safe_Str__Url__Host = 'localhost'

class SG_Send__Browser__Test_Harness(Type_Safe):
    config     : Schema__Browser_Test_Config          # auto-instantiated by Type_Safe
    persistence: Harness_State__Persistence            # auto-instantiated by Type_Safe
    api_server : Fast_API_Server        = None         # None when not yet started
    sg_send    : SG_Send__Browser__Pages = None
```

Type_Safe auto-instantiates nested Type_Safe fields. When you create `SG_Send__Browser__Test_Harness()`, the `config` field is automatically a fresh `Schema__Browser_Test_Config()` instance.

## Naming Convention

Following the existing codebase patterns:

| Pattern | Example | When to use |
|---------|---------|-------------|
| `Schema__*` | `Schema__Upload_Page` | Pure data objects (DTOs, page state captures) |
| `QA_*` | `QA_Transfer_Helper` | Utility classes with methods |
| `SG_Send__*` | `SG_Send__Browser__Pages` | Classes representing SG/Send concepts |

The page schemas should use `Schema__` prefix (matching `Schema__Browser_Test_Config` and `Schema__Harness_State`).

## Summary

| | @dataclass | Type_Safe |
|---|---|---|
| **Project standard** | No | Yes (mandatory) |
| **Mutable defaults** | Requires `field(default_factory=...)` | Handled automatically |
| **JSON serialisation** | Manual or `asdict()` | Built-in `.json()` |
| **Constructor** | Auto-generated | Auto-generated from annotations |
| **Methods** | Supported | Supported |
| **Composition** | Manual instantiation | Auto-instantiated nested fields |
| **None handling** | `str \| None = None` | Use `str = ""` (empty over None) |
| **Used in codebase** | 0 files | 15+ files (100%) |

**Action required:** When implementing the page object schemas from commits 2-5 of the architecture plan, use `Type_Safe` not `@dataclass`. The corrected examples above are drop-in replacements.
