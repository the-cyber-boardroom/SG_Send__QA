## What's Failing — 17 Failures in 4 Root Causes

### Root Cause 1: `#file-input` Inside Shadow DOM (9 failures)

**The problem:** In v0.3.0, the upload page was decomposed into sub-components. `<upload-step-select>` now uses `this.attachShadow({ mode: 'open' })` and the `<input type="file" id="file-input">` lives inside its shadow root. Playwright's `page.locator("#file-input")` cannot pierce Shadow DOM, so it times out after 5 seconds.

**Confirmed in code:** `upload-step-select.js` line 16: `this.attachShadow({ mode: 'open' })`, line 165: `<input type="file" id="file-input" ...>`

| Test | Priority | Error |
|---|---|---|
| `p0__combined_link::test_combined_link_upload_and_auto_decrypt` | **P0** | Timeout waiting for `#file-input` |
| `p0__friendly_token::test_friendly_token_upload_and_resolve` | **P0** | Same |
| `p0__friendly_token::test_friendly_token_format` | **P0** | Same |
| `p0__friendly_token::test_friendly_token_resolves_in_new_tab` | **P0** | Same |
| `p0__friendly_token::test_friendly_token_no_key_in_url_after_decrypt` | **P0** | Same |
| `p0__separate_key::test_separate_key_ui_flow` | **P0** | Same |
| `p0__upload__single_file::test_single_file_upload_flow` | **P0** | Same |
| `p0__upload__single_file::test_download_link_format` | **P0** | Same |
| `p1__upload__folder::test_upload_zip_and_gallery_view` | **P1** | Same |

**Fix approach:** Tests need shadow-piercing locators. For example:
```python
# Before (broken)
page.locator("#file-input")

# After (shadow-piercing)
page.locator("upload-step-select").locator("#file-input")
# Or use data-testid which is already in the HTML:
page.locator("[data-testid='file-input']")  # only works if Playwright pierces
```

Note: Playwright auto-pierces open shadow DOM for CSS selectors but NOT for `#id` selectors in `locator()`. The fix is either using `page.locator("upload-step-select #file-input")` (CSS piercing) or chaining locators through the shadow host.

### Root Cause 2: Download Pages Show "Transfer not found" (4 failures)

**The problem:** Tests create transfers via `transfer_helper.upload_encrypted()` (API), then open download pages in the browser. The browser's `send-download` component calls `/api/transfers/info/{id}` and renders "Transfer not found. The link may have expired."

The `send-download` component writes to light DOM (`this.innerHTML`), so `page.text_content("body")` correctly captures this error. The transfer ID exists on the QA server (the API tests prove this), so the issue is likely either:
- A timing/race condition in the JS fetch (unlikely given 4s waits)
- The `apiEndpoint` in `build-info.js` not matching the server URL
- A CORS issue blocking the cross-origin fetch from UI server → API server
- The `send-download` component not reading the hash fragment correctly at page load time

| Test | Priority | Visible Error |
|---|---|---|
| `p0__combined_link::test_combined_link_via_api_helper` | **P0** | "Transfer not found. The link may have expired." |
| `p1__sgmeta_magic::test_api_created_transfer_decrypts_with_correct_filename` | **P1** | Same |
| `p2__pdf_present::test_s_key_triggers_save` | **P2** | Same |

**Also affects (hidden by weak assertions):** Very likely the p1__navigation, p1__download__browse, p1__download__gallery, and p1__download__viewer suites are seeing the same "Transfer not found" but passing due to `or len(text) > 100` fallbacks.

**Critical investigation needed:** Open a browser manually, create a transfer via the API, and check whether the browse page actually decrypts. If it doesn't, this is a real product bug, not just a test bug.

### Root Cause 3: Inline JS Triggers False "error" Detection (2 failures)

**The problem:** The v0.3.0 download pages include a security patch that blocks friendly-token API leaks:
```javascript
return Promise.resolve(new Response('{"error":"blocked"}', { ... }));
```

`page.text_content("body")` includes text from `<script>` tags. Tests that assert `"error" not in page_text.lower()` fail because the string `"error"` appears in this inline JavaScript, not in visible page content.

| Test | Priority | Error |
|---|---|---|
| `p1__download__gallery::test_gallery_page_loads` | **P1** | `"error"` found in body text (from inline JS) |
| `p1__download__viewer::test_markdown_content_displayed` | **P1** | Content not found — shows "Loading transfer info..." (also related to Root Cause 2) |

**Fix approach:** Use `page.locator("main").text_content()` instead of `page.text_content("body")` to exclude `<script>` tag content, or use `page.inner_text("body")` which only returns visible text.
